# 机载电子硬件设计保证指南

# 1.0 介绍

随着与飞机安全性相关的电子硬件的复杂性不断增长，对飞机的安全性及鉴定产生了新的要求。这些要求的提出是考虑到由于硬件设计中的错误而使飞机功能更易于受到不利影响。硬件设计中的错误的出现是由于硬件复杂性的不断增加而造成的。因此，在设计和鉴定过程中，在谈及硬件设计错误时，有必要采取一致且可以校准的方式。

由于机载电子硬件变得更加复杂，本文件中描述的应用及程序的使用会促进技术发展以及经验积累。本文件根据批准的RTCA/EUROCAE程序修改和评审。

# 1.1 目的

本文件意在协助组织机构为机载电子硬件提供设计保证指南，如在特定环境下安全执行预定功能等。本文件应适用于现行的、新的、改进的技术。本文件的目的是：

1)、规定硬件设计保证目标  
2)、描述这些目标的基础，帮助确保正确理解本指南  
3)、描述这些目标，允许开发符合本指南和其它指南的方式。  
4)、提供设计保证指南，满足设计保证目标  
5)、允许灵活地选择程序以满足本文件中的目标，包括改进，如当的新程序技术出现时。本文件建议了应该执行的活动，以满足设计保证，而不是描述详细的设计实施。

该文件中用于生成指导文件的原理是一个基于正由电子硬件执行的系统功能的由上至下的观点，而不是一个由下至上的网点，或者是基于执行该功能的特定的硬件元件。自上而下的方法对于提到的安全调和错误更加有效，它使通知系统和硬件设计决定更为简便，使校验过程更为有效。例如，校验应在最高一级的系统、组件、分组件、元件或硬件项中执行，这样可以使硬件项符合要求，并且满足检验目标。

# 1.2 范围

本文件指南通过惯用准则和鉴定后产品的提高为机载电子硬件的设计保证提供了指南，以确保连续适航性。它是在表明符合鉴定要求的基础上开发出来的，用于运输类飞机和设备。  
本文件中的部分章节适用于其它设备。

本文件还描述了系统寿命周期和硬件设计寿命周期之间的关系，以帮助了解系统和硬件设计保证之间的内部关系。本文件中不要求对寿命周期的完整描述，包括系统安全性评估和校验，以及飞机鉴定过程。

鉴定问题只对于硬件设计寿命周期描述。硬件的生产、测试和维护能力相关的方面只有在与硬件设计的适航性相关时才提到。

# 1.6 复杂性考虑

虽然不同级别的“复杂性”，如简单的、复杂和高度复杂，都用于描述电子，但是并未对这些类别之间的区别作以严格规定。对复杂性差别的定义是以灵活性和通过确定性方法完成可接受的验证范围的难度等级为基础的。

硬件复杂性应分级检查，在集成电路、电路板和LRU级对复杂性（包括可能不能对其进行测试的编址功能）进行检查，如多用途器件中未使用过的方法，后续设备中隐藏的潜在状态等。

只有当把与设计保证级相关的确定性试验和分析综合起来可以保证功能性性能在所有可预定的工作条件下正确且无反正行为时，硬件项才可以标识为简单。

如果一个硬件项不能标识为简单，那么应标识为复杂。完全由简单项构成的硬件项自身也可能是复杂的。包含器件，如 ASIC 或 PLD 的硬件项，如果满足本章节中描述的对简单项的规定，则可以被视为是简单的。

对于复杂的硬件项，鉴定机构应同意建议的、在硬件设计寿命周期早期提供设计保证的方法，以减轻程序风险。

对于简单的硬件项，不需要大量的设计过程中的文献。对于简单的硬件项，必须执行校验和配置管理支持程序并且编制文件，但是不需要大量的文件。这样，设计简单硬件项的管理费用就会降低，符合本文件。本文件的主要影响主要是在设计复杂的硬件项方面。

# 2.2 系统安全性评估过程

有三种系统安全评估过程：功能性危险评估（FHA），初始系统安全性评估（PSSA）及系统安全性评估（SSA）。这些过程用于确立适用于系统开发保证过程系统安全目标，以及确立获得安全性目标的系统功能。

SSA 过程应把安全目标转换成系统和设备安全性要求。这些要求应体现系统和设备功能和结构的基本安全性目标和安全性特征。SSA 过程和系统开发过程把这些安全性要求分配给硬件。

有五个系统保证级：A～E级，对应于五类故障状况：灾难性、危险性/非常严重，不严重、以及不影响。表2-1把硬件设计保证级和五类故障状况关联起来，提供对硬件故障条件以及各自设计保证级别的的定义。最初，每个硬件功能的硬件设计保证级由SSA过程确定，使用FHA标识潜在的危险，然后PSSA过程把安全性要求和相关的故障条件分配给在硬件中执行的功能。

在整个硬件设计寿命周期中，在安全性、系统和硬件过程中会存在迭代反馈，以保证硬件的设计和建立满足分配给硬件的系统安全性、功能性和性能要求。

# 2.3.3 随机硬件故障的质量评估

与硬件的随机故障不同，设计错误或某些类型的差错都是不可以预见的，而且都有可能以公共模式故障穿越冗余界线。应对要使用的冗余管理技术和定量评估方法进行选择，这样就可以排除潜在的公共模式故障和混乱结果。

虽然很难定量评估，但是可以通过对定量安全评估方法的实际应用来有效地评估由于设计错误和混乱而造成的的安全性风险。分析技术，如故障树分析、公共模式分析以及功能性故障方法和效果分析，都是基础的定量方法，可以用于寻址硬件设计错误和混乱。更详细点说，这些方法可以确定设计错误和混乱的潜在的影响，并且确定可以使用哪种方法进行排除。使用这些方法，硬件安全评估可以用来确定要使用的硬件设计保证战略，在硬件设计过程中可以迭代使用，定量确定由所选择的战略带来的获益。

# 3.2 过渡准则

用不同的子项在不同的设计阶段开发一个硬件项很具挑战性，要求提供能够有一种方法对设计过程提供合理量的控制，以便在前一个过程完全完成之前，管理启动下一个过程的风险。过渡准则，定义为用于评估从一个过程移动到另一个过程的最小数据，可以用在关键的过程点上。计划过程中的分析应确定使用过渡准则。计划中规定的每对过程步骤之间不必确定准则。过渡准则的选择应考虑到对安全的影响。例如，在鉴定可信度时，在执行功能校验之前，需要首先用文件确证对该功能的要求，该功能的实现也需要在配置管理之下进行。

过渡准则应在硬件计划中用文件确证。过渡准则的使用不包含任何特殊的生命周期模式或阻止快速建立模型和同时进行的工程等开发战略。

# 5.1 要求捕获过程

要求捕获过程标识和记录硬件项要求，包括那些由建议的硬件项结构、技术选择，、基础的和可选择的功能、环境和性能要求派生的要求，以及由系统安全性评估施加的要求。设计过程中，自额外要求变为已知要求起，该过程可以重复。

# 5.1.1 要求捕获目标

要求捕获过程的目标是：

要求被标识、规定和用文件证明。这包括从PSSA分配的要求和从硬件安全评估中派生的要求。

# 注释

对硬件要求的验证结果的可追溯性在第6节中讨论。理想方式是在要求捕获过程中确立可追溯性方法。

产生的派生要求被反馈到相关过程中。

在相关过程中提供要求忽略和错误以寻求解决方案。

# 5.1.2 要求捕获活动

要求捕获活动形成一个重复，该过程可以保证这些要求与设计实施、系统要求和软件要

求的一致性。

要求捕获活动的指南包括：

分配给硬件项的系统要求应用文件证明。这些可以包括标识要求，如功能和性能，以及结构考虑，如隔离、内部测试、可试验性、外部接口、环境、试验和维护考虑、功率和物理特征等。

应标识与硬件项相关的、来自PSSA的安全性要求。这些包括：

硬件要执行的功能的设计保证级

对故障或功能丢失的可能性要求

所选择的、用以满足功能分配的硬件结构和功能性安全属性，如2.3.1节中所描述

应标识由生产过程、标准、过程、技术、设计环境和设计指南引起的设计制约

应确定实施所需的派生要求。应专门标识从含有安全要求的安全性评估中派生的要求。

# 注释

派生要求应考虑到下列状况：

用以确保较高设计保证级别功能的专门制约可以经受较低设计保证级别的功能，如《低设计保证级接口功能》中所示

典型的、全范围的数据值数据输入范围，以及数据字中高/低状态位或控制寄存器

启动复位或其它复位状态

电源电压和电流命令

与时间相关的功能的性能，如过滤器、集成器和延迟

可能发生的状态机器转换，不管是否是预期的

在正常或恶劣条件下信号定时关系或电气状况

信号噪音和串扰

异步逻辑电路中的信号失灵

控制未使用功能的专门制约

派生要求应回馈到 SSA 过程，这样就可以评估对系统要求的影响

要求数据应使用量化术语编写成文件，允许容差。但不包括对设计和校验方案的描述。

应把该过程中发现的要求忽略或错误提供给系统开发过程。

要求，包括为满足PSSA要求而产生的，应可以到下一个更高层级的要求。应尽可能沿层级标识和追踪派生要求。

# 注释

分配的硬件安全要求的系统级校验可以在要求捕获过程中发生。派生硬件要求的校验在第6.1节中描述。

# 5.2 概念设计过程

概念设计过程中产生高级设计概念，可以对这个高级设计概念进行评估，以确定设计实施过程中的潜在结果是否满足要求。这可以通过使用功能块图、设计和结构描述、电路卡装配纲要和底盘略图等来完成。

# 5.2.1 概念设计目标

概念设计目标是：

硬件项概念设计是根据要求开发的。

产生的派生要求回馈给要求捕获或其它相关过程

把要求忽略或错误提供给相关过程，寻求解决方案

# 5.2.2 概念设计活动

概念设计活动指南包括：

产生对硬件项的高级描述。包括：

与安全性相关的结构制约，包括对地址设计错误和功能、元件过应力、可靠性和强度缺陷的必要考虑；

对软件或其它系统元件的任何实施制约的标识

应标识主要元件。应确定它们对硬件安全性要求的影响，包括未使用功能的影响

派生要求，包括接口定义，应回馈给要求捕获过程

要求忽略或错误应回馈给相关过程以寻求解决方案

对要提供的可靠性、维护和试验特性进行标识

注释

建议对已满足的概念设计目标取得相关方面的一致性同意。一般来说，通过设计评估来获得一致同意。

# 5.3 详细设计过程

详细设计过程把硬件项要求和概念设计数据作为详细设计的基础，产生详细的设计数据。

# 5.3.1 详细的设计目标

详细的设计目标包括：

详细设计是从硬件项要求和概念设计数据中开发出的

派生要求回馈给概念捕获过程或其它相关过程

把要求忽略或过程提供给相关过程寻求解决方案

# 5.3.2 详细的设计过程活动

详细的设计活动指南包括：

硬件项详细的设计数据应以要求和概念设计数据为基础产生。这包括装配和互联数据、元件数据、HDL、试验方法和硬软件接口数据。

注释

在详细设计过程中，校验方法非正式地用于促进技术决定的制定。例如，设计

参数分析，如逻辑定时和参数校验可以为基础设计决议提供信息。

根据需要采用结构设计技术。这些包括建立功能安全监测、确定功能和安全监测之间的相异点，排除会对安全性产生影响的设计错误，以及容错设计。

试验特性应设计成允许校验安全性要求。

注释

以这样的方式开发设计是非常重要的：使安全特性不仅在硬件设计寿命周期过

程中可以校验，而且还可以作为验收试验和场地返回维护试验的一部分进行校验。

可以对未使用的功能时行评估，以标识潜在影响

对设计、安装或硬件项工作的制约，如果不遵守，会影响硬件项的安全性，这一点应该标识

详细设计过程中产生的派生要求应回馈给概念或其它相关过程

详细设计过程中发现的要求忽略和错误应提供给相关过程寻求解决方案

# 5.4 执行过程

执行过程使用详细设计数据产生硬件项，投入到试验活动中。

# 5.4.1 执行目标

执行过程的目标是：

使用具有代表性的制造过程执行硬件详细设计，产生硬件项。

硬件项的实施、装配和安装数据是完整的。

把派生要求回馈给详细设计过程或其它相关过程

把要求忽略和错误提供给相关过程解决方案

# 5.4.2 执行活动

执行活动的指南包括：

硬件项应使用设计数据和生产产品的预定资源产生。包括采购、配套、建构、检验和试验。

执行过程产生的派生要求应回馈给详细设计过程或其它相关过程

执行过程中的忽略和错误应提供给相关过程寻求解决方案。

# 5.5 生产过渡过程

在这个过程中，应对加工数据、试验设施和通用资源进行检查，确保对于生产的可用性和适用性。生产过度过程使用执行和校验过程的输出，把产品推入生产阶段。

# 5.5.1 生产过渡目标

这个过程的目标是：

确立基线，包括支持硬件项连续复制所需的所有设计和加工数据。

标识与安全性相关的加工要求，用文件记录，建立加工控制

把派生要求回馈给执行过程或其它相关过程

把错误和忽略提供给相关过程寻求解决方案

# 5.5.2 生产过渡活动

生产过渡活动指南包括：

应从配置的设计数据中准备加工数据

应检查加工数据的完整性和与配置的设计数据的一致性

# 注释

本文件不对加工制造文件添加任何条件。

在生产过渡过程中生产的任何变化或改进都应评价，以确保符合生产要求，特别是安全性要求。不符合用户或认证要求的任何变化都应取得相关方面的批准。

应明确规定从属于安全性的加工要求，以便在生产过程中控制。

应明确确定开发验收试验准则所需的数据

把标识的忽略或错误提供给相关过程以寻求解决方案

# 5.6 验收试验

验收试验证明：加工的、改进的或修理的产品功能符合单元的主要品质（认证基础）执行。这些主要品质是根据工程选择，并且指示产品能够满足开发时的预定要求。

# 注释1

制造产品时的配置控制不是由验收试验活动要执行的一项功能。配置管理计划，如本文件第7节所述，应描述应用计划是如何执行该活动的。

本文件的范围不包括确定验收试验准则，包括通过/失败条件。产品活动，包括验收试验，被视为不在本文件范围之内。

# 注释2

验收试验不用于校验每个产品单位的所有要求。

子项试验可以做为验收试验的一部分进行。

验收试验准则应确保：

1、标识电气试验  
2、必要时标识环境筛选试验  
3、验收试验提供满足安全性要求所需的这些设计方面的范围。应标识试验没包含的与安全性相关的硬件项或子项，并且提供其它保证方法。这些方法可以包含分析、设计控制、统计过程控制或其它适用方法。

# 5.7 系列生产

该过程不在本文件范围之内，但是简单地描述了影响设计保证的因素，使寿命周期完整。  
该过程根据生产数据和要求程序化地生产硬件项。

考虑包括：

1. 生产过程变化或设计管理保证：变化不会对当前安全性、谁或要求符合性产生负面影响。

# 注释

除了本文件中建议的指南外，第11.1.1节包含了与以前开发的硬件的改进。元件废弃问题请参看11.2页。

2. 与变化相关的所有文件的升级都应根据批准的配置管理计划执行。

# 6.0 确认及验证程序

此部分描述了确认及验证程序。不管系统对硬件项目的要求如何，确认程序可保证所派生的硬件要求是正确的和完整的。验证程序可保证硬件的执行满足包括派生要求在内的所有硬件要求。

# 6.1 确认程序

不管系统对硬件的要求如何，此处讨论的是通过使用主观及客观组合程序，确认程序如何保证派生要求的正确性和完整性。确认程序可在硬件项目确认前或之后进行，但是，确认通常是在整个寿命设计周期期间进行的。

注1：经验表明，对研制和确认加以注意，即可以在研制周期的早期阶段确定细小的误差，减小随后在设计中出现的问题或硬件性能不足的问题。

此处所讨论的确认程序并不是为了使系统所指定的要求有效，因为对这些要求的确认是假定在系统程序某一部分发生时进行的。此外，并不是对所有硬件项目的派生要求必须进行确认。

对于设计结果会影响系统安全和功能（用于系统其它部分）的要要求应归类为派生要求，并应对之进行确认。此外，对随后的设计任务会造成限制的设计结果和假设也应作为派生的要求进行确认。

需要进行确认的派生要求应按照指定给硬件项目的系统要求进行确认。对于不可能达到更高水平的派生要求应按照设计结果进行确认。

注2：具有执行专门功能的单独电路电源的设计结果，会产生派生要求以指导电源设计。这些派生的要求应包括以故障条件为基础的安全要求，此故障条件是因为由从电源模块上接收电源的电路的故障所引起的。对这些要求应进行确认。

另外一个变为派生要求的设计示例是，外围仪器存储地址的分配。此要求的制定通常没有任何基础，但是，一旦确定，它们就会限制随后的设计任务使之符合那些分配。此派生要求不需要进行确认。

# 6.1.1 确认程序目标

用于硬件派生要求的确认程序目标是：

1). 根据硬件项目派生的硬件要求应进行验证，证明其正确性和完整性。  
2). 评价派生要求对安全的影响。  
3). 将遗漏和错误反馈给合适的过程中以解决问题。

# 6.1.2 确认程序的活动

可通过一组行动进行硬件目标确认，如，检查、模拟、原型机、模型、分析、维护经验、工程评价或测试研制及执行。

确认程序活动指南包括：

1.) 应确定要进行确认的硬件派生要求。  
2.) 对于第一条中所确定的每个要求，应确定确认完成标准，并满足以下要求：

a. 通过检查，分析或试验分等级对每个要求进行了确认。

b. 对每个要求进行检查、分析或试验的方法适合于要求确认，特别是安全要求。  
c. 与每条要求的确认相关的检查, 分析或试验结果应是正确的, 并对实际的和期望的结果之间的差异进行了解释。如果没有对期望的结果预先进行确定时, 则确认行动的结果应符合要求, 特别是安全要求。

注：确认完成标准是以要求，安全考虑，操作模式或执行为基础的。

3.) 应评价派生要求对安全的影响。  
4.) 不管系统对于硬件项目的要求如何, 应对硬件派生要求的完整性进行评价。此程序的目的是, 当所有的属性均确定为必需的并且所有必需的属性已经被确定后,一整套要求应是完整的。  
5.) 应按照系统分配给硬件项目的要求, 评价硬件派生要求的正确性。根据此文件的目的, 当所规定的要求没有模糊之处并且没有错误时, 此要求才是正确的。  
6.) 在硬件派生要求与确认行动及结果之间应确立可跟踪性。

7.) 遗漏要求及错误应反馈给合适的部门以解决问题。

# 6.2 验证程序

验证程序可保证硬件的执行满足要求。如此计划中所规定的，验证由检查，分析及试验组成。验证程序还应包括对结果的评价。

注1：硬件设计的安全要求，采用通过硬件执行才能符合的安全要求形式。

此部分对在硬件设计中要使用的验证程序提供了指南。如硬件验证计划中所规定的，验证程序可用于任何层次的设计中。从安全角度考虑，在设计过程的各个阶段使用验证程序在提高可信度方面的中益的，可以取消设计误差。与附录A中的规定无关，一些设计保证水平要求满足验证程序目标。

此处未对软件验证，软件/硬件整体验证和系统整体验证程序做出规定。但是在这些程序中进行的硬件验证要求是硬件验证确认的一种方法。

可使用类似的、分析的、新的设计试验或重复初始验证的一部分对已验证过的配置进行再验证。

注 2: 推荐使用文件规定验证程序之外的非正式的试验。但是, 不需要在配置管理控制中保存其步骤和结果, 它们在设计的早期过程在探测和取消设计错误是非常有效的。只有等验证信用正式后, 才可用于测试。

# 6.2.1 验证程序目标

验证程序的目标是：

1. 为硬件执行满足要求提供证据；  
2.) 在硬件要求与执行之间，验证步骤与结果之间建立了可跟踪性；  
3.) 规定了验收试验标准，并且此标准是可执行的，与硬件功能的硬件设计保证水平是一致的；  
4.) 将遗漏的和错误的内容反馈到其合适的过程中进行解决。

# 6.2.2 验证程序中的行动

可以通过一些组合方法，如检查、分析、研究及试验来满足验证程序目标。验证计划用文件说明了验证行动（用于证明符合要求）。

验证行动包括：

1. 确定需要验证行动的要求。但是并不是要在每个等级都对这些进行验证；也可以在更高等级上对这些要求进行验证。  
2.) 应选择并执行相关验证方法，如试验、模拟、样机研究、分析及检查。  
3.) 在要求、执行、验证步骤及结果之间确立可跟踪性。可跟踪性应与硬件所执行的功能的设计保证水平一致。除了安全考虑以外，在跟踪性方面，不要求具体到部件上，如电阻器、电容器或门。  
4.) 应执行验证分析以确定验证步骤的完整性，这些分析包括：

a. 应通过检查、分析或试验在每个等级上对每个要求进行验证；

b. 对每个要求的检查、分析或试验同样适用于要求验证，特别是安全要求；  
c. 与每项要求的验证相关的检查、分析及试验结果应是正确的，并且应对实际的和预计的结果之间的差别进行说明。当预计的结果未按检查、分析时的状况预先确定时，验证行动的结果应与要求一致，特别是与安全相关的要求。  
5.) 验证行动的结果应形成文档。  
6.) 应将遗漏的和错误的内容反馈到合适的过程中进行解决。

# 6.3 确认和验证方法

此部分描述了一些可应用于确认及验证的一些方法。

# 6.3.1 试验

试验的方法是使硬件设备能对一些刺激或一系列的刺激产

生正确的反应。这些试验的示例包括硬件设备上的功能性试验，系统台式试验，系统确认时的设备试验及飞机上的试验。

这些试验可能会通过人工完成，自动完成或特殊试验设备完成。这些试验同样会利用硬件内部仪器试验性能，如验证程序中的自检测。

当通过预定的环境进行硬件项目的试验来完成特殊的验证要求不可行时，可以使用其它的验证方法。

这些试验会在各种硬件设计过程中执行。用于鉴定信用所执行的试验要求已配置好的设备。系统集成或软件/硬件集成试验结果同样可用于试验信用。

试验指南包括：

1. 确定要通过试验进行确认或验证的每一条要求。环境鉴定试验要求是这些要求的一部分。  
2. 应确定每项试验所用的试验刺激、序列及试验条件，如设备的环境温度和所施加的电压。  
3.) 试验执行之前应确定通过/失败准则及记录结果的方法。  
4.) 应记录试验设备的整体鉴定及每一个的校准日期。  
5）应记录要测试硬件设备的配置特性。  
6.) 应将失败试验反馈到合适的程序中以便于解决。

# 6.3.2 分析

分析方法是一种详细的、可重复的、可分析的方法，用于评价特殊的硬件设备特性以证明符合特别的要求。分析方法的示例有应力分析，设计范围分析，普通模式故障分析，最坏情况分析及试验覆盖范围分析。

注：随着硬件设计复杂程度的增加，使用计算机化工具有很大的益处，如模拟验证要求执行设计。

这些分析包括详细的功能、性能、可跟踪性及硬件设备功能的安全执行检查，以及与机上系统或设备内的其它功能的关系检查。单独进行分析或与其它的验证方法一起进行分析可以提供这些要求被正确执行的证明。这些分析应以设计程序、维护经验或其它可获得数据库所提供的数据为基础。

对于可视化电路操作及较高电平的功能性操作来说，模拟的是一种特别重要的设计分析工具。模拟可以用于分析当硬件参数改变时所产生的影响，用其它的验证方法很难做到这一点，可以建立因为这些变化所引起的设计错误对安全的影响减小的信用。因为这些结果依赖于所使用的模式及条件状况，所以单独使用模拟结果而不提供其确认证据，是不能在以鉴定信用为目的试验中使用的。

有关分析的一些示例如下：

1）热分析：热分析可检验当暴露于热工作环境下时，设计执行是否满足要求。  
2. 应力分析：应力分析可检验在要求的工作范围内，部件是否满足降低了等级的相关准则的要求。  
3.) 可靠性分析：可靠性分析确定了设计是否满足产品的可靠性要求。  
4.) 设计范围分析: 设计范围分析可检验在假定的部件变化情况下, 设计执行是否满足功能性要求。  
5.) 相似性分析：相似性分析比较了以前鉴定的系统特性及用途。  
6.) 模拟分析：模拟分析比较了模拟结果及期望的结果。

# 6.3.3 检查

检查是一种用于评价计划、要求、设计资料、设计原理或设计执行是否合格的量化的方法。

如相关计划中所规定的，应通过硬件设计寿命周期进行检查。应在确认和验证计划中对鉴定信用中所要使用的所有的检查做出规定。

检查指南包括：

1.) 参加人员必须具备执行检查的知识;  
2.) 硬件检查结果可批准或否定硬件设计寿命周期过程之间所进行的活动的转换;  
3.) 检查结果应形成文档，包括所做决定和所要采取的行动的安排。

# 6.3.3.1 要求检查

要求检查是确保要求的可接受性的方法。要求检查需要在相同的检查中确定确认和验证这两者的目标。

在首次要求检查之后发生的要求的变化应经受与首次检查相同的或者等同的检查过程。本次检查的意图并不是确认分配给硬件的系统要求。

要求检查指南包括：

1. 每项要求均应是不模糊的、可验证的、对其分级水平均应有详细的说明，并且不能与其它的要求冲突；  
2.) 分离的要求应符合系统要求或其所分离出来的原来的要求;  
3.) 这些要求应与 SSA 一致;  
4.) 对所分离的安全要求应做出规定并反馈回 SSA;  
5）这些要求应与相关的硬件设计要求标准兼容；  
6）这些要求应与可获得技术的性能与限制相兼容；  
7. 部件要求，如，性能、温度范围、降级及筛选等，应与安全和可靠性要求一致；  
8. 应对试验性能、维修性能及硬件部件的制造性能做出决定；  
9）软件/硬件接口要求应做出规定；  
10.) 根据计划中所规定的准则，这些要求应可以追踪到下一个分级水平上；  
11.) 所分离出的要求应接受相关执行限制，这些限制不会在较高等级水平中进行验证；  
12.) 遗漏的和错误的地方应返回到适当的步骤中。

注1：下列问题对评价要求的完整性可提供帮助：

a. 是否对所有的上一级的要求进行了考虑；  
b. 是否对所有适用的标准和指南进行了考虑；  
c. 是否包括了硬件功能和接口；  
d. 体系结构是否包括了所有方面；

e. 是否对所有的硬件执行要求做了充分的规定；  
f. 是否在安全评价中包括了所有被禁止的要求特性；  
g. 是否对工作环境做了充分的规定；  
h. 是否对假设及限制进行了考虑；  
i. 此执行是否避免了硬件中现存的及相似的任何已知的问题；

注2：下列问题对评价要求的正确性会提供帮助：

a. 所有的要求是否均是依据上一级的要求；  
b. 所有的要求是否依据系统指定给硬件项目的要求；  
c. 这些要求中用“什么”而不是“多少”；  
d. 这些要求是否明确；  
e. 这些要求是否能够实现；  
f. 这些要求是否能够验证；  
g. 是否对功能模式做了说明；  
h. 这些要求是否与安全评价一致；

i. 这些假设和限制是否与分离的要求一样被正确的做了规定；

# 6.3.3.2 设计评审

设计评审是一种用来确定设计数据及执行是否满足要求的方法。设计评审应按硬件设计寿命周期期间，每次计划中所规定的要求进行。这些示例如，概念上的设计、详细设计及执行评审。当确保设计正确性的电位为最大值时，应对包括了硬件部件设计的分级设计，如ASIC及电路卡组件、设计评审进行考虑。

设计评审指南包括：

1.) 应对所有的要求、分离的要求及设计要求进行正确的确定；  
2.) 应对环境要求进行确定；  
3.) 应对安全和可靠性要求进行确定；  
4.) 应对设计数据中的安全要求进行明确的规定；  
5. 设计应能执行，测试并维持；  
6.) 应对新的制造技术进行评价；  
7.) 计划中所列的部件选择准则应能满足;  
8.) 按要求，设计应是可追踪的；  
9.) 遗漏的和错误的部分应反馈到合适的部分中。

# 7.0 配置管理过程

配置管理过程是为了使配置项目能始终如一的重复，重建所需要的信息，如果需要改型即可在可控制模式下修改配置项目。此部分描述了硬件管理目标及支持此目标所要采取的行动。

# 7.1配置管理目标

1.) 应对配置项目特别规定并形成文档；  
2.) 确保配置项目的一致性和精确重复;  
3.) 提供了一个可确定及跟踪配置项目修改的可控制的方法.

# 7.2 配置管理目标

配置管理行动指南包括：

1.) 配置项目应是专门规定的、形成文档的并且是可控制的。这些内容包括但不限于，硬件、硬件设计表示、工具或用于信用鉴定和基准的其它数据项目；

2.) 应确定基准；  
3.) 应对问题进行特别规定,跟踪及报告;

4.) 应保持对这些改变的变化控制和跟踪性;  
5.) 归档,检索及配置项目的发布应是可控制的.

可使用各种方法来满足配置管理目标及行动,以下章节提供了可接受方法行动指南:

# 7.2.1 配置确定

配置标识行动的目标是对每个配置项目进行明确的标记，以便于建立一个控制基础和配置项目参考。

此指南包括：

1. 应为数据项目建立配置标识；  
2.) 应为每个配置项目、配置项目中的每个单独控制的部件、由鉴定机构批准的符合计划要求的组成产品的配置项目的组合等, 确立配置标识.

注：对如 ASIC,配置的 PLD,印制线路板及黑匣子等部件的详细说明由配置管理计划确定.

3.) 当 COTS 部件及预先研制的硬件项目在基准中使用之前, 应为它们确定配置标识.  
4.) 当每个配置项目在新的基准中使用之前,应为它们确定配置标识, 以便于其它的数据项目参考或用于产品制造.

# 7.2.2 基准的确定

基准确定的目的是确定以后所采取行动的基准，并允许将其作为控制配置项目之间可跟踪性的参考。

1.) 应为信用鉴定所用的配置项目确定基准：

注：可确定中间级基准以帮助控制硬件组织。

2.) 一旦确定了基准，则它应经受控制变化步骤。  
3.) 当从一个已确定的基准中研究一个分离的基准时，应遵从控制变化指南。  
4.) 如果在研制新的基准时, 如果需要寻找信用鉴定用于所采取的行动或与以前的基准设计相关的数据, 则此新的基准应可跟踪到它所分离出的旧的基准。

注：基准可以是配置项目、以前鉴定的硬件项目或COTS部件。

# 7.2.3 问题报告、跟踪及校正行动

问题报告、跟踪及校正行动的目的是记录问题并确保正确的部署及解决方法。这些问题包括：不符合计划和标准；寿命周期过程输出失效；产品的反常反应；及工具和工艺过程的不足及失效。问题报告应在不超过基准（从此基准中可获得鉴定信用）确立的时间内执行。

此指南包括：

1.) 对于所报告的每一个问题均应包括在问题报告中;  
2.) 在问题报告中应对受到影响的配置项目进行确定；  
3. 要求校正行动的问题报告应使用控制变化行动；  
4.) 所有完成了的问题报告应包括结束此问题报告所采取行动的说明, 包括需要执行校正行动的完整的数据项目的变化;  
5.) 并不是所有的问题报告都需要终止以得到确认, 但是, 应对所有的问题报告进行评价, 而且那些已确定的项目应是安全的, 其鉴定影响应结束;  
6.) 问题报告系统应跟踪问题报告状态，包括它们的批准和调用。

# 7.2.4 变化控制

变化控制行动的目的是确保变化的记录、评价、解决方法及批准。变化控制应按照配置管理计划执行，并且应在基准确立（从其中可得到鉴定信用）之前开始。

# 此指南包括：

1. 通过防止非授权机构所做变化，变化控制应保持配置项目的完整性；  
2.) 变化控制应确保, 被评定后的变化可用于确定配置标识是否需要更新;  
3.) 变化控制条件下的配置项目的变化，应被记录、批准并跟踪；

注 1: 问题报告与变化控制相关, 因为对所报告问题的解决会导致配置项目出现问题;

注2：通常认为，早期执行的控制变化可以帮助过程行动的控制和管理。

4.) 变化控制应确保所做变化的原因的可跟踪性;  
5.) 变化控制应确保对变化所产生影响的评价，以确定在过程输出上变化的影响，并应确保输出数据的更新；

注1：所有的或部分的过程行动应从其注受到影响的部分开始进行报告；  
注2：应认识到工具制造、工艺过程或外部部件的变化均会对设计造成影响；  
6.) 变化控制应确保将有影响的过程反馈回去。

# 7.2.5 发布、存档及检索

发布行动的目的是在配置管理控制条件下放置数据，以确保只在授权的数据才能在其它的行动中使用。存档及检索的目的是，当需要复制、重建、再测试或修改产品时，确保与产品相关的数据项可以被检索。

此指南包括：

1.) 在制造之前，配置项需被标识并发布，并应确立其发布授权；  
2.) 与产品相关的数据项应可以从其批准之处检索到，如研制组织或公司；

注：变化控制数据及问题报告数据均为数据项的一部分。

3.) 数据保留步骤应可以满足适航要求并允许进行改型;  
4.) 应确立步骤以确保鉴定机构所要求的存储数据的完整性；

a. 确保无非授权变化；  
b. 选择存储介质；  
c. 保持存储数据的可获得性。例如, 通过介质存储寿命相兼容的频率使用或刷新存档的数据;  
d. 确保不会因为一个单一的事件导致存档数据丢失。例如，可以在独立的不受影响的存档个体上保存复本。

# 8.0 过程保障

过程保障可确保符合寿命周期过程目标，按计划完成所有的活动，并指定所有的偏离要求。此部分说明了过程保障的目标及支持那些目标的行动。其目的不是为了暴露专门的组织结构。

过程保障行动应可以独立的达到，以便于从客观上评价寿命周期过程，确定偏离并保证校正行动。

# 8.1 过程保障目标

过程保障的目标是为了确保：

1. 寿命周期过程符合已批准的计划；  
2. 所产生的硬件设计寿命周期数据符合已批准的计划；  
3）应确保对硬件计划及标准的偏差能进行探测、记录、评价、批准、追踪及解决。  
4.) 应确保硬件寿命周期过程转换准则的满意度与批准的计划一致。

注：对于执行上述1至4项，审核是一个有效的方法。

5）应进行检查，以确保硬件项目的制造与设计数据一致。  
6.) 应产生对过程保障行动的记录，包括设计行动完成的评价证明在内；  
7.) 如果适用，其应用应确保转包商所使用的过程与硬件计划一致。

# 9.0 鉴定联络程序

鉴定联络程序的目的是，在整个硬件设计寿命周期期间，确立用户与鉴定机构之间的通讯及相互理解，以便于对鉴定过程有所帮助。鉴定联络应按照硬件计划内所述完成。第4部分，PHAC，第10.1.1部分，附录A中的表A-1给出了此程序输出摘要。此外，联络行动可包括设计方法说明，以便于及时批准，双方的协商应考虑符合鉴定基础的方法，设计方法的批准，数据批准的方法，及鉴定机构要求的任何检查及试验见证。在项目完成时，应出具设计过程摘要，并应在硬件完成摘要中说明生成输出及硬件项目状态，见第10.9部分。

# 9.1 符合性方法及计划

申请人建议了一种硬件符合性方法。PHAC 确定了建议的符合性方法。此指南包括：

1.) PHAC，应将硬件验证计划及其它要求的数据提交给鉴定机构，用于当设计变化对程序的影响为最小时，进行检查；  
2.) 应对由鉴定机构考虑到硬件鉴定所做计划而确定的观点进行规定；  
3. 可在鉴定机构得到PHAC的协议；  
4.) 如计划中所说的, 在设计和鉴定周期内与鉴定机构的联络应继续, 并且对鉴定机构提出的观点应及时解决。

在一些程序中，鉴定联络不由设备制造商提供，而是由主机厂或由在设备制造方面起支持作用的其它用户决定。此关系应在PHAC中规定，与鉴定机构的联系应通过鉴定申请进

行。这点应由鉴定申请人负责，以确保将数据提供给鉴定机构。

当设备中嵌入的一些硬件项目是从转包商处获得的时，鉴定计划应确定那一个数据由转包商提供，那一个由申请人提供。

对于申请人来说，在最高级鉴定计划中包括PHAC和验证计划及其它相关计划是可以接受的。

# 9.2 符合性证明

申请人应提供硬件设计寿命周期过程满足硬件计划的证据。鉴定机构的检查可替代申请人的设备或申请人所提供的设备。申请人安排这些检查并在需要的时候取得硬件设计寿命数据。

申请人应：

1. 解决鉴定机构在检查时所提出的问题；

2.) 给鉴定机构提交硬件完成摘要，10.9 部分及最高级图纸，第 10.3.2.2.1 部分要求的内容；

3. 提交鉴定机构要求的其它数据或符合性证明。

# 10.0 硬件设计寿命周期数据

此部分说明了硬件设计寿命周期数据项，它们可以在硬件设计寿命周期期间产生，以提供设计保障及符合鉴定机构要求的证据。按照设计保障证据的要求，鉴定机构所需要的寿命周期数据的范围、数量及详细情况，会根据某些因素而变化。这些因素包括鉴定机构对机载系统的适用要求，指定的设计保障水平，硬件复杂程度及使用经验。对设计保障证据的详细情况也会在PHAC中做出规定和记录，并由鉴定机构批准。

第11章中的附加的考虑及附录B中对A级和B级功能的设计保障考虑会增加寿命周期数据。

如第 7 部分所规定的，按照硬件设计保障水平，附录 A 指出了要研制的硬件设计寿命数据，验证程度及所应用的数据控制类别。

1.) 硬件设计寿命数据特性：

a. 明确性：信息/数据应明确的写出，只应有一种解释；  
b. 完整性：信息/数据应包括：需要的和相关的要求及说明材料，有标号的图纸，确定的术语，测量单位；  
c. 可检验性：应由一个人或一种工具就可以检查信息/数据的正确性；  
d. 一致性：信息/数据应是无冲突的；  
e. 修改：当保存信息/数据时，可对其结构和变化进行完全的、一致的进行；  
f. 可跟踪性：可以确定信息/数据的源；

此部分的说明，不是为了暗示一种特殊的数据打包方法及形式，或包内硬件寿命周期数据的结构。例如，可在单个的文档或多个文档中对所有的计划、标准及步骤进行说明。

2.) 应在 PHAC 中对数据打包方法、形式及结构提出建议，并应在程序进行早期获得鉴定机构的批准；  
3.) 在机载系统或设备的整个使用寿命期间，批准的信息/数据应是可以检索和得到的。

# 10.1 硬件计划

硬件计划说明了硬件鉴定、设计、确认、验证、过程保障和配置控制所使用的过程、步骤、方法和标准。

# 10.1.1 为硬件鉴定所做的计划

PHAC 确定了达到本文件目标及获得鉴定机构对系统所含硬件项目鉴定批准所要使用的过程、步骤、方法及标准。一旦批准，应执行鉴定申请人与鉴定机构之间在过程和行动方面的协议一PHAC，并应产生能证明满足硬件鉴定要求结果。

PHAC 也可是另一计划的一部分，如机载系统鉴定计划。

# PHAC应包括：

1. 系统概述：此部分提供了要使用硬件的机载系统的概述，包括系统功能说明、系统故障状态、系统结构、硬件及软件功能配置说明，及现存系统文档参考。  
2.) 硬件概述: 此部分说明了硬件功能、硬件项目、结构、要使用的新工艺、任何自动防故障装置、故障容限、要使用的冗余及分隔技术。  
3.) 鉴定考虑：此部分说明了鉴定基础、建议的符合性方法和每个硬件项目的设计保障水平。此部分同样提供了，以硬件安全评价为基础的硬件设计保障水平分配判据基础，及其在机载系统内的使用，包括如第 2.3.4 中所述的潜在的硬件故障条件。如果适用，还应包括：FFPA 摘要或执行 FFPA 的计划。  
4.) 硬件设计寿命周期：此部分说明了所要施加的步骤、方法和标准，及为满足硬件设计保障目标所要执行的过程和行动。还说明了：行动、行动组合和次序、过程和行动之间的关系、转换标准、责任、工具使用、在硬件过程之间、硬件过程与系统及软件过程之间提供反馈和互动的方法。对于程序计划和标准，可参考此部分中适用的计划、政策、标准、步骤和偏差。  
5.) 硬件设计寿命周期数据: 此部分说明或参考要研制和提交的数据, 或证明符合本文件和计划目标的数据。  
6.) 附加考虑: 此部分说明了附加的考虑。包括: 对以前研制硬件的使用, 包括对再使用数据的参考, COTS 使用, 产品使用经验, 第 11 部分所述的工具评定及鉴定或如附录 B 所述对 A 级和 B 级设计保证的考虑。  
7.) 可替换的方法: 此部分说明了建议给程序所用的任何可替换的方法, 在本文件中可能未对此进行说明。应提供可替换方法是否可用的判定。  
8.) 鉴定计划：当硬件设计寿命周期数据提交给鉴定机构时，此部分确定了主程序发展中的主要的几个步骤及数据。

# 10.1.2 硬件设计计划

硬件设计计划说明了要使用的步骤、方法和标准，及硬件设计中要执行的过程和行动。此计划可包括在PHAC中，并可作为设计原则和设计标准参考。

硬件设计计划应包括：

1.) 硬件设计寿命周期：参考设计原则和标准及硬件设计寿命周期过程和步骤说明，以达到硬件设计保障水平目标。  
2.) 硬件产品说明：要达到的硬件规范的要求，可替换方法，计划的使用寿命及升级条件。  
3.) 硬件设计方法: 对硬件要使用部件的说明: 要求和规范方法、原理设计方法、详细设计方法、合成技术、执行方法、生产转换方法。如附录 B 第 3.1 中所述, 当考虑了 A 级和 B 级功能结构减轻但还未实施时就已写下此报告, 则应写明如何在设计过程中实现此决定。  
4.) 硬件设计环境：说明了要使用的硬件设计工具。  
5.) 硬件设计项目数据: 硬件项目设计数据标识应在硬件项目规范、文档、图号及部件号之前产生。  
6. 其它考虑：说明了计划的过程技术选项、使用和组装选项、产品包装和硬件装置选项。

# 10.1.3 硬件确认计划

确认计划说明了要施加的步骤、方法和标准及所要执行的硬件项目的分离要求，以达到本文件的确认目标。此计划可包括在PHAC中，并可作为要使用的确认标准的参考。

确认计划包括：

1.) 确认方法: 要使用的确认步骤、标准和方法的说明。方法中应包括分析、检查和测试。  
2.) 确认数据：对于硬件确认过程中要产生的证据的说明和标识；  
3）确认环境：对分析和试验设备、执行确认时的过程和行动的说明和标识。

# 10.1.4 硬件验证计划

验证计划中说明了要使用的步骤、方法和标准，及为了验证所要执行的行动，以达到本文件所规定的目标。此计划可包括在PHAC中，作为验证原则和标准参考。

验证计划包括：

1. 验证方法：对验证原则、步骤、标准和方法作了说明，以提供对硬件项目的完整性证明，包括COTS及未使用的功能。方法中可包括：分析、检查和测试。当使用附录B中第3.3中所述的进一步的分析方法时，包括适用于FFP的详细的方法说明，及适用的验证准则。  
2.）验证数据：作为硬件验证过程的结果，应产生证据标识和说明；  
3. 验证的独立性：用于确保验证独立性的方法的说明；  
4.) 验证环境：关于分析和试验设备、用于执行验证过程和行动的验证工具的标识及说明；  
5.）机构的责任：组织标识负责验证过程的执行。

# 10.1.5 硬件配置管理计划

硬件配置管理计划描述了用于满足本文件配置管理目标的 原则、步骤、标准和方法。

配件配置管理计划应包括：

1. 硬件配置管理方法：用于规定、管理和控制硬件及其寿命周期数据的原则、步骤、标准和方法的说明；  
2.) 硬件基准：用于确立设计和产品基准的方法和步骤，并提供了基准的可跟踪性；  
3.) 问题报告和解决：用于记录、跟踪和解决问题的报告的方法和步骤的说明；  
4.) 变化控制: 用于确定、控制、跟踪变化以控制数据项目的方法和步骤的说明;  
5.) 存储和检索：用于硬件设计寿命周期数据发布、存档和检索的步骤的说明。此说明应包括存档内容、格式、介质标准、原则、方法和准则。  
6.) 环境控制：用于确定和控制用于研制和验证硬件的步骤和方法的说明；  
7. 配置管理工具：用于配置管理过程和行动的工具和源的说明；

# 10.1.6 硬件过程保障计划

硬件过程保障计划描述了要使用的步骤、方法和标准，及用于达到本文件过程保障目标所要执行的过程和行动。

硬件过程保障计划应包括：

1. 过程控制：对执行硬件设计过程保障所使用的原则和步骤；

# 10.2 硬件设计标准和指南

硬件设计标准和指南应规定硬件设计、确认、验证、保障和控制过程的原则、方法和准则；及用于评价硬件设计结果可接受性和质量的控制过程。可能不要求提供标准，但如果申请人在计划中使用了标准，则它们就成为鉴定基础和项目计划的一部分。在计划中，就应将此标准和指南作为单一文档或多个文档提供。工具执行标准也可使用。

# 10.2.1 要求标准

在要求的获取过程中可使用要求标准，以确定研制要求的原则、步骤、方法、指南和准则。要求标准可包括：研制和确定要求的方法和准则、验证要求的方法和准则，用来确认要求和表示要求的标记；此指南主要是关于要求规定工具的使用，及用于为系统设计过程提供分离要求的方法。

# 10.2.2 硬件设计标准

硬件设计标准可在概念设计过程中使用，详细的设计过程可确定原则、步骤、方法、指南和准则，以用于研制和确定硬件的设计。

硬件设计标准包括：

1）硬件设计表示方法和符号；  
2. 设计规范和命名惯例；  
3. 设计方法指南；  
4. 硬件设计工具使用指南；  
5. 电子部件选择指南；  
6. 设计替换方法评价指南；  
7）自动防故障装置和故障容限设计概念指南；  
8.) 到“要求过程”的反馈方法的说明，及用于澄清要求的说明；

# 10.2.3 确认和验证标准

在确认和验证过程中可使用硬件确认和验证标准，以确定确认和验证硬件设计和执行的原则、步骤、方法、指南和标准。

# 10.2.4 硬件归档标准

硬件归档标准可用于确定用于产品数据保持和归档、研制和保持程序及项目归档的步骤、方法和准则。硬件归档标准可包括归档内容、格式和介质标准、准则、方法和标准。

# 10.3.1 硬件要求

硬件要求规定所要研发的硬件的功能、性能、安全、可维修性及可靠性要求。

这些要求应包括：

1. 分配给硬件的系统设计和安全要求；  
2.）硬件适用标准的标识；  
3. 硬件功能和性能要求，包括正常使用所需要求和应力限制；  
4.) 硬件可靠性和质量要求，包括与故障率、暴露次数和设计限制有关的要求；  
5.) 在硬件项目的整个使用寿命期间，硬件的保持和维修要求；  
6）硬件的可制造性和处理要求；  
7. 硬件的可测试性要求；  
8. 硬件存储和处理要求；

# 9）安装要求。

# 10.3.2 硬件设计代表性数据

硬件设计代表性数据提供了硬件项目定义，并由生产硬件设备所需的一套图纸、文件和规范组成。

下面章节规定了一些典型的硬件设计数据及其内容。数据类型、图纸和给定硬件设计的文档将根据硬件所包含部件的尺寸、复杂程度及数据而变化。

# 10.3.2.1 概念设计数据

概念设计数据是一些描述硬件项目结构及功能设计的数据，包括：

1.) 对高级别情况的描述，如方框图或HDL定义，对主要的功能进行了概述，并说明了这些功能之间的流程；  
2.) 机械结构, 描述了硬件项目的配置, 如表示外部包装的图纸或略图、印制线路板布置、连接器选择和位置及主连接线路。  
3. 从适航观点来看，特别重要的一些其它结构特性和分区。这些内容可包括：EMI、照明、冲击和震动防护、主要部件上的未使用功能（如人机接口），如人类工程因素，照明特性和显示器分辨率；

4.) 高级硬件项目的功能说明;  
5）硬件项目功能结构；  
6）初步硬件安全评价数据。

# 10.3.2.2 详细的设计数据

在详细的设计数据中描述了硬件项目执行需要的数据。根据硬件项目的分等级水平，此项内容包括高级别的图纸、组装图、连接数据、部件数据、HDL 硬件说明、可靠性数据、试验方法数据、所选部件中未使用功能列表、所采取的不危及硬件项目安全的行动、安装控制数据、及硬件/软件项目接口数据。下面对一些专门的数据进行了描述。

注：除了其它鉴定所要求的详细的设计数据以外，如技术标准条例，申请人应在PHAC中，向鉴定机构建议其它详细设计数据项的内容。

# 10.3.2.2.1 最高级别图纸

最高级别图纸专门确定了硬件项目，并确定了所有的组件，子组件，部件及相关文档。

# 10.3.2.2.2 组装图

组装图包括硬件组装所要增加的专门信息。

组装图应包括：

1.)尺寸；  
2.) 容差要求；  
3.) 冷却和安装信息;  
4.) 重量、重心及其它确保安全和正确安装所需要的信息。

# 10.3.2.2.4 硬件/软件接口数据

如要求规范中所确定的，硬件的性能将根据软件对硬件的配置，软件对硬件的校准或硬件与软件之间的互相作用而定。它可以保障硬件按照其要求研制和设计，并且正确生产，达到其设计目标。这些数据包括硬件检查、分析和测试步骤和结果。如附录B中所述，A级和B级可能会有超出此部分的附加的数据项。

# 10.4.1 可跟踪性数据

硬件的可跟踪性确立了要求、详细设计、执行和验证数据之间的关系，使配置控制、修改及硬件验证项更易于执行。

硬件可跟踪性数据应包括：

1. 分配给硬件的系统要求和要求之间的关系；  
2. 要求和硬件详细设计数据之间的关系；  
3）硬件详细设计数据和硬件制造或组装项之间的关系；  
4. 要求之间的关系，包括分离的硬件要求及详细的设计数据和验证步骤和结果；  
5）可跟踪分析结果。

# 10.4.2 检查及分析步骤

硬件检查和分析步骤确定了执行检查及分析所需的过程及步骤：

硬件检查和分析步骤应包括：

1.) 检查与分析的目的;  
2.) 参与检查的组织;  
3. 检查与分析的准则  
4）执行检查或分析的详细说明；  
5. 检查或分析的可接受性和完成准则

# 10.4.3 检查与分析的结果

硬件检查和分析结果是检查与分析完成并批准步骤和准则的证明。

硬件检查与分析结果应包括：

1. 检查或分析步骤标识；  
2. 数据项检查或分析标识；  
3. 检查与分析中人员的参与；  
4. 检查或分析的结果；  
5.) 检查或分析所产生的校正行为，如问题报告列表或行动项；  
6. 检查或分析的结论包括：对于检查，应提供一个量化的项目检查；对于分析，应提供一个项目分析的量化评价及分析数据；

# 10.4.4 试验步骤

硬件试验步骤确定了执行功能和环境鉴定试验的方法、环境和说明。

硬件试验步骤应包括：

1.) 试验目的;  
2.) 硬件试验设置、软件标识，每个硬件试验所要求的试验设备建立的说明；  
3）执行试验步骤的详细的说明；  
4.）试验输入数据；  
5）预期的结果，如通过/失效准则和试验所包括的要求。

# 10.4.5 试验结果

硬件试验结果是已批准步骤试验已经完成客观证明，，以支持硬件项目的验证。

硬件试验结果应包括：

1. 试验步骤标识；  
2.) 被测试项目标识;  
3. 执行试验的实际结果；  
4.) 执行并见证试验的工作人员标识, 如果可行, 应包括执行试验的日期  
5. 结果说明，包括分析结果或检查结果及实际试验结果。

# 10.5 硬件验收试验准则

此准则应包括：

1）所要测试的关键的特性；  
2.）每个关键特性的通过/失败准则；

3.) 任何试验的限制;  
4.) 关键特性和通过/失败准则的证明;  
5.) 需要用来满足安全要求的设计范围;  
6.) 在实际的试验步骤和相关试验结果基础上的，表明试验准则已正确执行的评价数据。

# 10.6 问题报告

问题报告是确认并记录硬件设计问题解决方法，处理不符合硬件计划和标准及硬件寿命周期数据不足的一种方法。

问题报告应包括：

1. 配置项目和处理行动标识；  
2.) 要修改的配置项目的标识，或要改变的处理的说明；  
3.) 问题说明，可以使问题更好的被理解和解决；  
4.) 用于解决所报告问题的正确行动的说明。

# 10.7 硬件配置管理记录

配置管理处理行动的结果被记录在配置管理记录中。这些记录可包括配置标识列表，基准或电子记录，变化历史报告，问题报告摘要，工具标识数据，归档记录和记录发布。

# 10.8 硬件处理保障记录

处理保障处理行动结果被记录在处理保障记录中。这些记录可包括检查或审核报告，会议时间，授权的处理偏差记录，符合性检查记录。

# 10.9 硬件完成摘要

硬件完成摘要是表明符合PHAC并向鉴定机构证明硬件项目已达到本文件要求的主要的数据。此可由系统完成摘要组成。硬件完成摘要应包括以下信息，如PHAC中所规定的：

1.) 系统概述；  
2.) 硬件概述；  
3.) 鉴定考虑;  
4）硬件设计寿命周期说明；  
5）硬件设计寿命周期数据；  
6）预告研制的硬件；  
7. 附加的方法  
8.) 可替换的方法。

对与批准的PHAC的差别应做出标识。此外，以下4点应做出标注：

1）硬件标识：通过部件号和版本，此部分确定了硬件配置和硬件项目。  
2.) 变化历史：如果适用，此部分包括硬件变化摘要，特别注意了因为故障影响安全问题所做的变化，并确定了因为以前的鉴定而对硬件设计寿命周期处理所做变化。  
3.) 硬件状态: 此部分包含了在鉴定时未解决的问题报告的摘要, 包括功能限制摘要。  
4.) 符合状态: 此部分包括符合本文件的状态, 及用于证明符合硬件计划中所规定的准则的方法的摘要。此部分同样确定了与硬件计划、步骤及本文件相比,附加的原则和偏差。

注：PHAC中所包含的数据在硬件完成摘要中不需要重复，这样就会加快鉴定程序。

# 11.0 附加的考虑

此部分对前面内容中未包括的附加的设计保障考虑提供了指南。这些考虑可在申请人

判断是否满足第二部分到第九部分的一些目标时使用。使用附加考虑应征得鉴定机构的同意。

# 11.1 以前研制的硬件的使用

此部分讨论了与以前研制的硬件相关的一些问题的考虑。此指南包括对硬件更改、飞机装置、应用环境、或设计环境和设计升级基准的评定。第11.2中包括了COTS部件使用及以前研制的硬件的特殊状况的指南。同时还应指明对以前研制的硬件的使用的配置管理和过程保障考虑。

在PHAC中应说明使用以前研制的硬件的意图。

# 11.1.1 对以前研制的硬件的更改

此部分讨论了对以前研制的硬件的更改。因为要求变化、误差探测、硬件或技术提高或获得上的困难都会引起这些更改。

对建议的更改所采取的分析行动应包括：

1. 系统安全评价过程输出检查；  
2.) 如果硬件设计保障水平增加，应用第11.1.4部分；  
3.) 应对所做变化的影响进行分析, 包括由于这些所引起的对更多变化区域的再次验证。此区域可通过信号流分析, 功能分析, 计时分析, 可跟踪性分析或其它合适方法来确定。

# 11.1.2 飞机装置变化

此部分讨论了在一定的硬件设计保障水平及专门的鉴定基础上，在新飞机装置中对已鉴定硬件的使用。当在新的飞机装置中使用以前研制的硬件时，应使用以下指南：

1. 系统安全评价过程对新的飞机装置进行了评估，并确定了硬件设计保障水平和鉴定基础。如果这些部分与以前内容相同或者对新飞机装置没有以前的要求严格，则不要求附加的尝试。  
2.) 如果要求对新装置进行功能更改，则应满足第 11.1.1 部分中的“更改和预先研制硬件”指南。  
3.) 如果以前的设计行为没有产生可以证明新装置安全目标的说明, 则应满足第 11.1.4 部分中 A 级设计基准。

# 11.1.3 应用或设计环境变化

使用以前研制的硬件会涉及到新的设计环境，或与其它的软件或硬件的综合，而不仅仅是与原始使用的结合。

新设计环境会在硬件设计寿命周期过程范围内，增加或减小一些行动。此指南包括：

1.) 如果新设计环境使用硬件设计工具, 则应使用第 11.4 部分“工具评价和验证”中的指南;  
2.) 当以前研制的硬件与不同接口硬件一起使用时，则应执行硬件接口验证。  
3.) 当以前研制的硬件使用不同的软件时，应对硬件/软件接口的再验证进行说明。

# 11.1.4 设计基准升级

以下指南将用于以下这些硬件项目，即，根据以前的应用已确定了其寿命周期数据在新应用中不足的那些硬件。此指南旨在帮助申请人可以使其以前以低的硬件设计保障水平所研制的硬件能得到鉴定机构的批准：

设计基准升级包括：

1. 当利用以前研制的寿命周期数据的优势时，应满足本文件的目标；  
2.) 硬件鉴定应以如系统安全评价过程所确定的故障状态和硬件设计保障水平为基础；

对于所做变化对以前应用的影响应做分析，以确定失效区域；

3.) 对以前研制的寿命周期数据应做出评价, 以确保硬件满足验证过程目标;  
4.) 相反的工程可用于再产生有缺陷的或丢失的硬件寿命周期数据，以满足本文件的设计保障目标；  
5.) 如果打算使用产品 的使用经验来满足本文件升级时的设计保障目标, 则应确定第 11.3 “产品使用经验” 中的指南;  
6.) 申请人应确定“符合本文件中PHAC”的策略。

# 11.1.5 附加的配置管理考虑

在配置管理过程中，对以前研制硬件的新的应用，应包括（除第七部分的指南以外）：

1.) 对硬件产品和寿命周期数据以前的应用及新的应用的跟踪;  
2.) 变化控制处理，可以管理通用项目不同应用的变化要求。

# 11.2. 商业上可得到部件（COTS）的使用

COTS部件被广泛用于硬件设计中，并且不需要对COTS部件设计数据进行检查。鉴定过程不会对单个的部件、模块、或子组件进行检查，因为这些内容均包括在被鉴定的飞机的功能中。因此，COTS部件的使用将通过整个设计过程来验证，包括如文件中所规定的支持过程在内。电子部件管理过程连同设计过程的使用，为COTS部件的使用提供了基础。

# 11.2.1 对于COTS部件的电子部件的管理

对COTS部件的电子部件的管理是与硬件设计和研制相关的一项重要的支持过程。电子管理过程应用于COTS电子部件。当此过程中涉及到商业和技术要素时，此部分仅处理影响鉴定的技术部分的因素。

通过确立以下几点确立鉴定信用：

1. 部件制造商可提供高质量部件生产的跟踪记录；  
2.) 在部件制造时已确立质量控制步骤；  
3.）对部件的成功操作提供有使用经验支持；  
4. 制造商已对部件进行鉴定，或通过附加测试进行鉴定，以确立部件可靠性；  
5）部件制造商可控制部件的质量水平，或通过附加的部件测试保障其质量水平；  
6.) 在打算应用的技术适合性基础上选择部件, 如部件温度范围, 电源或电压额定值, 或附加测试或其它的方式;  
7. 部件性能和可靠性应在连续的基础上监测，将需要增强的部分反馈给部件制造商。

# 11.2.2 COTS部件步骤

COTS 部件获得指南并不是本文件的目的，但是当它们对硬件设计保障有严重影响时，申请人应对获得要求反馈进行管理并解决。

主要的观点包括：

1.) 如本文件所要求的，COTS 部件设计保障数据的实际可获得性；  
2.) 根据生产批次，部件参数的变化可不做确定；  
3）电子部件技术的发展；  
4.) 成为不可获得的COTS部件。

# 11.3 产品使用经验

使用经验可用于以前研制的硬件和COTS部件的设计保障证明。使用经验涉及到任何以前的及现在的部件的数据的采集。从非机载应用中得到的数据不包括在内。

注：部件的大范围的及成功的使用可说明其设计是成熟的、无误差的，也可以证明该部件制造的可靠性。

# 11.3.1 产品的使用经验数据的可接受性准则

当使用经验数据用于设计保障中时，使用经验数据的相关性和可接受性应根据下面要素之一或几个来确定：

1.) 与应用、功能、操作环境和设计保障水平相比较, 硬件项目使用的相似性;  
2.) 以建议的硬件项目配置为基础的设计保障数据的范围;  
3.) 在使用期间发现的设计误差范围，已被消除、减轻、或分析并确定对所使用配置无安全影响；  
4.) 与操作相关的实际的故障。

注：PHAC应按照使用经验数据，专门确定应用部件的设计保障要素。

# 11.3.2 产品使用经验数据的评价

为满足上述准则，申请人应：

1. 根据工程分析，评价以前的应用、安装及环境与目标应用的相关性；

注：用于确定使用及使用极限适合性的数据可以在规范、数据资料、应用备注、使用公告、用户责任及正误数据中获得。这些信息源同时也说明了与硬件项目相关的功能，所以预定使用的机载部件应与以前的使用有关。

2.) 对安全评价过程有所影响的预定的用途也应有所评价, 包括由数据规定的设计误差影响的可能的缓和在内;  
3.) 对设计误差及其对安全评价过程的影响的任何统计均应进行评价。如果不能得到此统计，则应使用量化的评价。  
4.) 对问题报告进行评价。问题报告会表明使用经验对现在的配置所做的提高。这些问题虽然已经确定，但是并不是不变的，可能会因为结构方法或附加验证产生误差。

注：对于电子部件最基本的使用也可能会引起已经删除或取消的错误。

# 11.3.3 产品使用经验评价数据

（用于证明建议的应用的设计保障的）使用经验评价数据应包括：

1. 部件标识及其在机载系统中的预定功能。确定设计保障水平或 A 级及 B 级中所使用部件的功能, 部件保障的附加方法说明, 比如, 要使用的结构方法及附加的或先进的验证战略;  
2.) 使用经验数据采集和评价过程说明, 包括确定数据充足性及有效性的准则;  
3.) 使用经验数据，包括要考虑的详细的使用信息、变化历史、用于分析使用经验数据的假设及分析结果摘要；  
4.) 与预定使用相关的使用经验数据充足性判剧, 及要求的保障水平。

# 11.4 工具评价和鉴定

硬件和软件工具通常是在硬件设计和验证期间使用。当设计工具被用于启动硬件项目或硬件设计时，工具中的一个误差就会导致硬件项目中出现错误。当使用硬件工具来检验硬件项目时，工具中的一个误差就会导致工具对硬件项目或硬件设计误差探测的失败。在工具使用之前，应先执行工具评定。评定的结果及工具的鉴定应记录并保持。

工具评定和鉴定的目的是为了确保工具能够执行特殊的设计或验证行动。

# 11.4.1 工具评价和鉴定过程

在工具评价中评价了工具在设计寿命周期过程中的作用，及根据工具的作用和硬件功能设计保障水平，所要执行的鉴定行动。此评价指南是以流程图的形式表示的，并应用于设

计工具和验证工具。此流程图致使申请人，到工具的一些被限定鉴定的类别中，及其它的工具鉴定中。

工具评价和鉴定过程可应用于单个工具或一组工具。这些工具所包含的性能通常会超过特定的设计或特定项目验证行动的要求。只需要对特定硬件寿命周期行动中所需的工具的功能进行评价，而不需要对整个的工具进行评价。

这些工具通常是综合在一起的，并且可以在各种寿命周期阶段共享。如果在设计和验证阶段使用了相同的工具，除非在两个功能之间有分离保护，否则需要将这些工具作为设计工具进行评价。

注 1: 如果对给定工具的评价表明它的一些功能是用于设计中的, 而其它一些是用于验证的, 则应对单个的功能进行说明, 并应对每组的功能进行评价。

注2：此评价主要集中工具应用上。

# 附录B

# A级和B级功能的设计安全考虑

# 1.0 引言

实现A级和B级功能的硬件设计者所做的设计决定对设备的安全性有很大的影响。随

着设计安全级别的增加，需要验证给定的设计是否满足其安全要求，而该验证方法可能需要与设计安全方法部分一致或分层组合。申请者可以选择一个或多个方法，建议另外一个可提供安全设计的方法。

本附录给设计者提供了如何使用FFPA开发一个安全设计方法的指导以及用某些特殊方法作安全设计的指南。

# 2.0 功能性故障流程分析

FFPA 是一个结构化的、自顶向下的重复性分析。它标识了实现该功能的特殊设计部分，即装配、元器件和与每个流程有关的部件；以及确定有关的故障方式、符合安全要求的硬件结构。FFPA 还能标识那些实现所设计的 A 级和 B 级功能的装配、元器件和部件。

一个FFPA是以PSSA开始的。PSSA被用来标识系统级FFP；而FFP又可以分解成、分配给硬件FFP。

FFPA 的目的是标识单个 FFP，为的是：

1. 实现 A 级和 B 级功能的硬件可以用一个适当的设计安全方法进行（已在本附录中描述），也可以用鉴定机关认可的其它高级方法。  
2.) 对于 C 级和较低级功能的硬件实现, 本章所述的方法是可选择的, 即只能用本资料第 1 章至第 11 章所述的指导方法实现这些功能。

注：用不同的方法所实现的功能可用不同的 FFP 标识；或提供不同程度的设计明晰度。这两点通常很有用，因为总的硬件项目的设计安全可以通过使用多个设计安全方法实现。分解的级别随着每个 FFP 变化。

可以用通常的自顶向下的安全分析方法来完成分解，如故障树分析法。对每个后续级别的分解，可以用FFMEA图表和普通分析方法来补充。对每个系统级别的FFP，分解的级别可根据设计安全策略而变化。安全策略相应于设计思想和所设计硬件的纠错方法。

分解过程：

从系统级 FFP 到硬件级 FFP;

从硬件级 FFP 到电路级 FFP;

从电路级 FFP 到元器件级 FFP；

从元器件级 FFP 到零件级 FFP。

# 2.1 功能性故障路径分析方法

FFPA按如下方法进行：

1.) 对每个A级和B级功能，要根据硬件要求和该功能的系统FHA来标识功能及其设计安全级别。该功能可作为子功能的组合，而每个子功能有其自己的要求和相应的设计安全级别。这些子功能可根据需要进一步分解。  
2.) 对于每个 A 级和 B 级功能，确定实现其功能或子功能的方法，然后分析设计安全选项。用于实现该功能或子功能的可利用（或希望可利用的）的安全数据应该是完整的并且对设计安全策略（或所选择的策略）来说是可以接受的。如果可利用（或希望可利用的）的安全数据是完整的、正确的和可接受的，则无需进行进一步分解。  
3.) 对于不是 A 级或 B 级的 FFP，它们与 A 级或 B 级的相互关系应当用 F-FMEA（即普通方式分析或根据图表方式）来进行评估，从而保证不是 A 级或 B 级的 FFP 对 A 级和 B 级的 FFP 不会产生不利的影响。

该评估过程是可以重复的。如果对一个 FFP 来说没有可接受的设计安全方法，则其分解和评估过程被重复，或硬件功能的结构或实现被改变，直到设计安全的可接受方法被确定、可接受的安全数据被提供或每个 A 级和 B 级的 FFP 可以被提供。

FFPA的结果和所选择的用于硬件安全的方法要和飞机系统过程联系起来,如本资料2.1节所述。这些结果用来测定和有效地确定飞机级别,尤其那些与类似硬件项目的多交互系统

有关的设定仍然有效。

# 2.2 功能性故障通路分析数据

FFPA数据应当：

1). 标识那些从系统级来看对硬件项目有代表性的不规则的行为和功能性故障。  
2). 标识 FFP、异常行为和功能性故障的影响、设计层面上（在该层面上完成分析）的分解级别以及可接受安全数据（可获得）的类型和位置。  
3). 描述 FFP 之间的关系，确定其独立性及其它 FFP 和元器件的相互独立性。这样的关系可以用定性 FTA 或其它自顶向下分析、普通方式分析、F-FMEA 或根据图表来分析。这些关系描述可以识别那些相关路径、元器件和相互依赖性。  
4). FFP 与硬件要求（及导出要求）之间的的关系。

# 3.0 用于A级和B级功能的设计安全方法

本附录的目的并不是通过当前或将来方法去限制设计安全的实施。在本附录中讨论的方法可以用来满足上述过程的一个或多个目的，如资料第4章到第6章所述。

# 3.1 结构调节

可以使用结构设计特性来调节或抑制硬件设计和实现误差的不利影响，例如不同的实现、冗余、监视器、隔离、分割和命令/授权限制。作为PSSA的一部分，有些方法（如定性故障分析和普通方式分析）能够提供确定结构属性的范围的可靠手段。这些结构属性在调节或抑制硬件故障以及设计和实现误差时很有用。还有，该方法可以与硬件的FFPA方法结合使用（如附录B中第二节）并且用普通分析方式来确定特殊调节策略的适应性和硬件设计与实现误差的范围。例如，冗余通常在随机故障或失调范围内有帮助，但是选定普通方式，则冗余也可用来有效地调节设计和实现误差。

# 3.1.1 结构调节方法

结构调节是通过明确与所提出的硬件实现有关的 FFP、然后分析设计选项来进行的。这些设计选项是用来标明设计特性和调节 FFP 的作用的策略。对有关调节 FFP 的所有相关作用，所提出结构的总影响应该进行评估和说明。使用结构调节策略是为了引入某些有关的要求，这些要求的实施应该进行验证。尤其重要的是，这些结构特点会保护某些或所有已明确的 FFP 的有害影响。同时应该对附加的故障路径进行说明。这可以通过进一步的结构调节或另一个设计安全策略（已在本附录中描述）实现。

# 3.1.2 结构调节方法

安全评估过程决定了结构调整的可接受性。FFP应当首先标识所有的A级和B级硬件FFP（在这里结构调整被用于编辑）、标识所使用的方法以及调整的原则。是否充分，取决于在总的结构相似情况下对支持该调整的每个功能所做的评估，而结构的相似性会或多或少牵扯到结构调整的复杂程度。

普通方式分析会发现潜在的普通方式错误。而这些错误在需求、实现、制造和维护中会使调节无效。设计者也应该考虑到形成结构调节功能的硬件会产生潜在的随机故障，这些故障会使调节无效。支持调节功能的概率可利用性和调节失效的结果成反比例。而调节的失效会减小安全裕度。

所有的方法都应该在必要的功能之间保持和维持正确的操作和适当的独立性。任何需要消除、隔离或与其它普通方式效果相关的安全装置都应该以附加的结构调节形式或以其它设计安全措施方式（在本附录中作了规定）标明或组合。

当结构的定义完整时。决定不调节或不充分调节的A级和B级FFP硬件功能将用另外的设计安全方法（在本附录中规定）决定。例如，单个电路和元器件的部分结构调节可以用特殊的安全分析方法来标识和提供所使用电路和元器件的未调节部分的鉴定范围。

# 3.1.3 结构调整数据

为了保护硬件A级和B级FFP，结构调整方法的文件应提供安全评估数据、安全要求数据和溯源数据。安全评估数据是跟据硬件FFP的评估、特别是普通方式分析来解决硬件设计的结构调整。

结构调整数据包括：

1. 由结构方法保护的A级和B级硬件FFP的标识。  
2.) 结构方法和由该方法提供的有效原则范围的描述。  
3.) 普通方式界面和适用于该结构的普通方式设计的原则。  
4.) 未调节和未充分调节的 A 级和 B 级硬件 FFP 的标识将由其它设计安全方法完成。  
5.) 有关结构调节机制的功能性操作和必要的设计属性的要求。  
6.) 用来满足安全要求的调整机制包括软件，如分割软件、监控软件和其它不同的软件。应该给系统过程和软件开发过程提供这些机制和安全软件要求。  
7.) 对于完成结构调整的任何硬件，应注意一般的故障率数据和潜在的故障暴露评估数据。  
8.) 安全要求与安全评估数据以及设计鉴定数据相联系的溯源数据。

# 3.2 产品维护经验

第11.3节给机载硬件的使用和对产品维护经验数据进行正确评估提供了基本指南。对于使用以前开发的硬件作为一部分设计的A级和B级功能使用附加的设计安全是必要的。可以用以下方式提供设计安全。

# 3.2.1 产品维护经验及方法

在完成第11.3节评估后，由硬件实现的FFP应当就维护使用经验进行分析。使用者或设计者标明维护经验数据并确立：维护经验数据证明该硬件的重新使用已经在以前的硬件使用中充分演示。

# 3.2.2 维护经验数据的重新分析

当维护经验数据的分析完成之后，如果A级和B级FFP的硬件功能没有被演示、没有充分地演示或维护过程中没有经验可用，那么这些硬件功能应使用另一种设计安全方法或能用来演示该功能的附加鉴定标识来确定。

# 3.2.3 产品维护经验数据

适用于保护A级和B级FFP硬件的产品维护经验数据应当包括：

1.第11.3.2节产品维护经验评估数据。  
2.) 通过维护经验和对维护经验数据效率的判断, 为 FFP 的标识提供设计安全。  
3.) 为了 FFP 的标识，光有维护经验数据是不充分的。试验环境、试验过程、分析和结果的标识综合起来才能完成 FFP 设计安全。  
4.) FFP 的标识和操作条件不是由维护经验表现的。维护经验要求附加的结构调整或高级的检验方法。  
5.) 溯源数据（如 10.4.1 所描述）表示了维护经验数据和鉴定之间的明显关系。该鉴定为 FFP 提供了设计安全范围。

# 3.3 先进的鉴定方法

要达到另外的设计安全可信度并获得证据，要使用先进的检验方法，例如：元件分析法、正常分析法、特殊安全检验分析或用户提议并由有关当局接受的方法。

设计安全的先进检验方法可使用和扩展附录B中第2节所提出的FFPA方法的范围。FFPA方法可以逐步应用到设备级、电路级和元件级，从而确定A级和B级FFP的硬件实现。来自FFPA的数据用来决定所提出的设计安全方法，该方法适用于硬件电路、元器件和包含在A级和B级FFP中的元件。

这里有三种方法，现描述如下：

1.) 基本分析法。该方法提供了一种自底向上的完成硬件检验的测量方法。在 FFP 内的每个功能元件都被标识并用试验鉴定，它们应满足 6.1 节规定的鉴定目标。该方法也可标识用其它合适的方法来确定问题的范围。  
2.) 特殊安全分析法。该方法着重于暴露和校正从系统安全角度出发对硬件输出的不利影响。硬件输入空间和输出空间的适当安全敏感性部分可通过分析来确定。硬件输入空间的敏感部分被激活；输出空间不但可用于安全敏感性要求的功能鉴定，而且可用于异常情况。观察输出空间的方法可以通过分析预先标出。但该分析是通过传统的安全分析技术来完成的。  
3.) 正式方法。正式方法是使用逻辑数学和离散数学的技术用于计算机系统的规范、设计和鉴定。这些技术可以用来实现硬件设计寿命周期不同过程中的推理。

使用者也可以提出不同于本资料的其它先进的鉴定方法。

# 3.3.1 基本的分析

基本分析法可用来表示 FFP 已经由有关的试验鉴定。它提供了设计错误已经由分开复杂 FFP（分成基本元件）进行了消除。FFP 的分开在设计者产生错误时完成。该方法为鉴定过程提供了一种方式，从而确定了鉴定的范围和完善，因而最适合细节设计可见并且在配置控制下的情况。在 ASIC 和 PLD 中情况如此，因为在那里各种功能在相同的设计安全级下被完成；否则不同设计安全级的各项功能被隔离或分开。可应用的电路或元器件的每种功能元件都被标明并用各种过程来鉴定所选功能的正确性（见 6.1 节）。基本分析法适合于一个整元器件或组件而无需考虑各种 FFP 实现的个数。但是如果为各种不同 FFP 的隔离、独立和分离提供一个原则，则该基本分析法适合于部分元器件或组件。

注：当基本分析法在一个PLD功能中运用时，应该包括规定的内容和PLD特性的应用；对于未规定的元器件，则可以使用分开的方法（如前所述）进行。

该方法标明了用适当的途径解决有关问题的范围。不经过这种分析的鉴定过程可能会使某些电路的试验不充分。据检查，这样的不充分是由于缺少试验过程、硬件要求不清除或不完善、有未使用的电路、未初始化的电路和文件功能。该方法能测试 FFP 中元件的鉴定情况并决定与每个元件有关的鉴定范围是否很全面。如果元件的鉴定范围不完善，则需另加鉴定或适当的活动。

申请使用者应提出在哪个设计层面上规定元件以及如何分析鉴定范围。

# 3.3.1.1 基本分析方法

该分析法开始时要规定一套标准，即要考虑硬件设计安全级别、硬件技术和实现硬件的细节。

这些标准应包括：

1.) 在硬件设计的适当级别上零部件的标识和定义。  
2.) 每个零部件都应当在鉴定范围内进行鉴定。

这些标准适应于分析检验活动，从而确定检验范围内完整标准是否通过有计划的检验来获得。如果这些标准无法获得，那么正在测试的零部件可以通过适当的激励激活，从而观察对正在试验下监控的信号的的影响。

注：当该过程检测硬件本身的试验时，可以检测出试验过程中的失效。对该试验失效的分析可以提供另外的可信度以及试验是否充分的证据。执行新的或修改过的试验能够发现硬件中的故障。

# 3.3.1.1.1 选择基本分析标准

适用的基本分析标准应当根据硬件元件的类型及其复杂程度和元件可标识的操作功能进行逐个选择。该分析可以表明：所有低级初始单元（如计数器、寄存器、多路转换器、加法器、运算放大器和滤波器）都被正确测试，或者所有相互连接的初始单元都被正确测试并

且满足检验范围标准。该试验过程的分析标准可以通过对零部件功能操作进行评估来获得；同时与其它硬件器件结合起来考虑，从而完成更高层次的硬件功能。

注1：例如，如果一个元件是一个用于延时的模  $\mathbf{n}$  计数器，则该试验过程可以适当地选择输入数据以验证启动时计数、关闭时停止计数。同时以正确的速率计数，在规定的时间达到模  $\mathbf{n}$ ，然后翻转。没必要证明试验过程已到达构成计数器的各个门电路或触发器。

作为使用互连原始器件作为一个元件的例子，可以用算术逻辑单元构成元件，如计数器、加法器和控制逻辑。算术逻辑单元（ALU）可以模仿地表示原始单元集体构成了 ALU，但是在基本分析法中使用的检验过程应使用适当的输入数据证明 ALU 可以完成其功能。

零部件不需要在硬件设计者规定的设计层次上定义。如果该设计被明确表示为组合逻辑或机械状态控制门，则可以使用门级分析法。

注2：分析由设计者在下面规定的层次级别实现（如在门电路级或晶体管级）是没有必要的，因为这种分析类似于在装配语言级或二进制级上的软件分析。这些较低抽象级通过对硬件鉴定试验的基本分析或对设计后模拟成功地进行评估（如果必要，可以根据11.4节作为合格检验工具）而隐含地进行了说明。

一个 ASIC 或 PLD 可能包含专有的数据功能。这些功能提供其内部设计的能见度，因此没有在用户手册中叙述。数据功能可以在基本分析中被看作是 COTS 元件。COTS 硬件方面在 11.2 节和附录 B 的 2.2 节中作了说明。这些数据功能的检验标明它们与其规范或数据产生者提供的描述是一致的。该试验应当根据环境来进行，该环境使试验结果可以观察。

注3：这里的目的不是要阻止使用设计资料库以利于建立新的功能；设计资料库实际使用是为了鼓励尽可能减小硬件错误的几率。

因为ASIC或PLD是从HDL的高级描述合成而来，所以该分析标准是建立在代表硬件的高级行为语言代码基础之上。然而，因为HDL表达的综合实施可能包括平行的逻辑结构和非时序暂时方面，所以合成输出应该包含在分析完成后的结论中。也应该对合成器进行评估。

# 3.3.1.1.2 完成基本分析

基本分析法是在下列试验环境中一个或多个条件下所完成的鉴定试验的基础上进行的分析：

1. 用电路实现安装在目标组件上的功能通道试验。  
2.) 在独立样机上进行的试验。这样的试验对一个 ASIC 或 PLD 来说是典型的。  
3）制造检验试验。

注：因为生产试验通常不建立在这些要求基础上，所以生产检验试验在应用基本分析法时会受到限制。

4.) 一个设计后的模拟，典型地是为了已经被评估的 ASIC 或 PLD。如果必要，该模拟可以用作检验工具（见 11.4 节描述）。

如果被分析的试验过程与正在使用的基本分析标准有关，并且该试验过程可用于第6节中所述的硬件功能的鉴定，则基本分析法本身可以用模拟来测量所取的完整性。如果分析的试验过程可以从硬件或单独样机的内部电路试验中获得并且用模拟进行测试；如果转换过程对基本分析之一的精度作了检验，那么试验激励和预期的结果可以被转换、用作模拟器。用来完成基本分析的模拟器应该能证明，它决定在应用中包括的每个类型的元件都能满足分析标准。

# 3.3.1.2 基本分析结果透析

基本分析法能发现某个硬件元件未被检验，这就意味着需要另外的检验过程或需要去

掉未检测的元件，或者调节由于结构方面所引起的异常行为。硬件元件未测试可能是由下列原因引起：

1.) 漏掉某个试验测试过程。如果试验简单地没有根据有关标准（附录B中3.3.1.1节有规定）对硬件项目的某个元件进行测试，那就有可能引起漏测。如果在功能要求中存在“不仔细”但硬件项目设计中有意产生重复响应，则也可能产生漏测。在这些环境下，可以补充或改变试验过程和情况。而且，鉴定相应要求的试验能力也应当进行评估。  
2.) 要求不充分。要求应当能改变或另外获得的要求也能改变。对新的、修改过的要求，应当进行另外的鉴定试验及其分析。  
3.) 有些功能未使用。硬件项目可能会在其目标应用中包含未使用的功能，例如数据库功能内有未使用的子功能或试验结构只用作元器件级的验收试验。这样的功能应该表明已与其他功能隔离或表明不会有异常行为、对安全有不利的影响。这可在硬件内或安装时标明：那些未使用的元件已经与其它部分断开、失效。如果在某时，未使用的某些功能要启用，则基本分析法失效部分可被重新访问，前提是这样的功能要被标识为“未完全检测”。  
4.) 元件没有不利的安全影响。分析表明：元件异常行为的影响不会对飞机或其乘客产生不利的安全影响。通过记录元件异常行为的影响及其分析，可以解决这个问题。

# 3.3.1.3 基本分析法之寿命周期数据输出

基本分析法之寿命周期数据输出应该：

1.) 通过基本分析法确定 FFP 的标识问题，同时提出在设计层次的哪个级别上元件被规定以及如何分析以及适当鉴定，这些都是鉴定范围完成标准的一部分。这包括在 PHAC 或硬件鉴定计划内。  
2.) 在分析中描述所使用的方法并标识 FFP；在完成分析的设计层面上标识级别。  
3.) 要保证在 10.4.1 节中描述的溯源数据能表明检验过程和基本分析法中元件的明确关系。  
4.) 标识鉴定试验情况和作为分析结果所增加或改变的要求。  
5.) 标明 FFP 所达到的鉴定完成级别（FFP 由分析法处理），包括分析误差的鉴定。鉴定试验或要求以及验收原则的修改未能解决分析误差的问题。

# 3.3.2 特殊安全分析

在应用时，特殊安全分析法可以延伸到硬件FFPA概念。这是通过对所选择电路和元器件进行更深层的分析来完成的。延伸的FFPA用来获得和确定这些电路和元器件内部操作的特殊安全要求。这些获得的安全要求由鉴定试验处理，下面将讨论这个问题。

特殊安全分析是基于这样的概念：只有当特殊的输入激励源处于试验条件下时，可能的潜在设计错误才会影响到硬件项目的输出。因此，为了充分激励和暴露有关的安全隐患，应当标识必要的安全操作输入数据的子集，然后将子集中的相当一部分包括在鉴定试验内。如果导致不安全输出条件的特殊异常情况不出现，则在这些试验的执行期间应对项目的输出进行评估。特殊安全分析用来将鉴定试验中施加的一系列输入条件联系起来，这样就不必要处理许多潜在的输入试验情况。

# 注：该实现也可能将输入数据集和许多条件相联系，因此要使该实现来测试极限之外的输入是不可能的。

特殊安全分析方法也可以用来确定电路和元器件功能的不可调整性。在这些功能中只有部分的结构是可以调整的。在这种情况下，另外的特殊安全分析法可能是一种有用的和有效的方法，这种方法可以确定到达安全范围还需要什么设计安全。

特殊安全分析法同样适用于COTS硬件或定制电路和元器件，因为它能使用这些电路和元器件的用户指导数据而不是详细的内部设计数据。通过结合用户指导数据与FFPA方法的详细应用，特殊安全分析法能够成功地确定电路和元器件使用的安全灵敏范围和有关的内

部 FFP（这里有必要强调排除设计错误）。该信息可用来成功获得电路和元器件鉴定试验，当该试验进行中，可以极大地暴露电路和元器件设计错误，同时提供改正和调整该错误的机会。从系统角度看，该错误会对硬件产生不利的影响。

# 3.3.2.1 特殊安全分析方法

一旦用特殊设计安全分析法处理的电路和元器件被选择，即采用另外的FFPA对它们进行详细测试。该分析法确定了哪些电路和元器件的功能属于已经标识的A级和B级功能（这些功能使用那些电路和元器件）。这时，逐个检测每个电路和元器件并在其功能临界处用该电路和元器件的实际功能完成包含在更高级的A级和B级FFP中的硬件功能。

注：在电路和元器件用户指南数据中有充足的信息，指导你用电路和元器件功能完成更高级的硬件功能。如果有电路和元器件内部功能的足够信息可利用，则可对此作充分的评估。如果没有充足的信息可利用，则不能作此评估。因此，应该使用另外的方法，或与此方法结合起来使用。

当根据电路和元器件的实际使用标识了有关安全敏感功能后，下一步就是进行更详细的分析。该分析将确定那些电路和元器件功能的特殊安全敏感性、及未调整的属性。电路和元器件功能将通过特殊安全鉴定条件来进行详细的分析。这些鉴定条件可使用F-FMEA技术获得并生效，从而确定安全敏感性的特殊功能属性；同时更进一步确定电路和元器件内部组成A级和B级FFP功能的任何特殊异常情况。

通过上述特殊安全分析活动获得的鉴定条件要与下列指导联系起来才能完成鉴定电路和元器件的特殊安全分析标准（包含在A级和B级FFP中）。指南包括：

1.) 标识有关功能的输入空间。根据标识的安全敏感性功能属性和异常情况确定有关的输出空间合格或失败，同时形成相当的类别，以提供有关输入空间的必要范围。  
2.) 标识相对显著的检测方法和每个有关功能的输入空间激励方法。

# 注：可以使用特殊工具和实现特性来保证可观查性和可试验性。

3.) 规定试验环境以检查潜在的错误源和相互依赖性。

注：元件级功能应该在可行的最高综合级别上进行测试。在较高综合级上的测试通常提供最佳的错误源范围，例如不稳定、相互依赖和潜在的相互交叉功能作用。试验应当用相当的类别进行。试验应当解决关键的逻辑决定、算术、定时、状态转换和实时属性等问题。

# 3.3.2.2 特殊安全分析决定

特殊安全检验完成标准应当由所有适用的电路和元器件的特殊安全分析来建立。由分析和或由鉴定本身所发现的任何错误都应该由下列方法之一去解决：

1.) 更改设计以纠正错误。  
2.) 增加结构调整。这可以从相关 FFP 去掉该结构而排除该故障。  
3）增加适当的试验。

# 3.3.2.3 特殊安全分析数据

当特殊安全分析资料应用于A级和B级FFP中电路和元器件时，应当以安全评估数据、安全要求数据、鉴定过程和结果以及溯源数据的形式提供该资料。为了安全要求和特殊安全分析，鉴定过程应该是可跟踪的。特殊安全分析数据应当包括：

1.) 标识将由特殊安全分析法所处理的电路和元器件。  
2.) 每个电路和元器件所在的 A 级和 B 级 FFP 的标识。  
3.) 适合于电路和元器件的部分结构调整的标识。在电路和元器件中的设计安全由特殊安全分析法提供。  
4.) 对每个适用的电路和元器件，都有安全敏感性功能标识。  
5.) 对于每个标识的安全敏感性功能，都有安全敏感性属性和有关异常情况标识。

6.) 处理可应用电路、元器件、内部功能、功能属性和异常情况的检验条件。  
7）处理输入依赖性和待检验输出空间情况的鉴定情况。  
8.) 检验过程和结果。

9.) 将检验过程和硬件安全检验条件与特殊安全硬件分析数据相联系的溯源数据。

# 3.3.3 正式方法

正式方法指的是在规范、设计和计算机系统的构建中所使用的逻辑和离散数学中的方法。

注：本章资料来自“用于计算机系统及软件鉴定的正式方法规范和分析指南，第二卷：操作指南。”1997年5月，NASA-GB-001-97。正式方法的更详细叙述在该书中举例进行了说明。

正式方法的应用可分为两大类：描述型和演绎型。描述型方法使用正式的规范语言对有关要求和其它设计品提供清楚的、毫不模糊的描述。演绎法使用一种原则：要求对所有情况进行列举。同时还要进行一步步推理。此外，每一步推理必须要允许适度的干预。最严格的正式方法使用推理技术来证明有关要求、其它设计方面或实现复杂和关键系统。正式方法的目的是为了减少在评估中依赖人的直觉和判断。也就是说，演绎法可以减少一个论点对某个计算的依赖性。而该计算原则上能用一个工具检查，用一个可重复的操作代替评估过程中的固有主观性。

有几个区域，使得正式方法的使用在设计过程中提供另外的保证。虽然正式方法适用于整个设计过程，但设计安全的增加可以通过预定的目标获得。下面列出了一些主要概率：

1.) 正式方法可应用到研制寿命周期的不同阶段。一般来说，正式方法的应用在寿命周期的早期阶段最有效，特别是在满足要求和高级设计期间。  
2.) 正式方法可用于整个设计过程或用于特殊元器件。FFPA 用来确定哪个 FFP 可用正式方法分析。有关并行复杂通讯的协议和实现容错功能的硬件可以用正式方法来进行有效的分析。  
3.) 正式方法可以用来检验系统功能或可用来建立特殊性质。虽然正式方法一般与“正确检验”有关，即能使元器件满足其功能规范，但它们也可应用到最重要的特性中去。通常，验证某个设计不出现某些不需要的特性比证明它具有所有的功能更重要。

正式方法的实际应用特别要求有工具支持。所使用的工具应当进行检查，必要时需满足11.4节所规定的要求。

# 3.3.3.1 正式方法的工艺

正式方法的应用是用正式语言表达开始的。要求的规范用作一个重要的描述功能。它用清楚的概念为文件编辑、通讯、以及确定系统的作用和特性提供了一个基础。此外，要求的规范还为计算和预测系统行为提供了一个基础。一个需要分析的元器件的正式模式是用正式语言组建的。该模式是用所选择的正式逻辑的规则根据正式要求的陈述来分析的。该模式的特性由需要完成的正式分析的方式确定的。

元器件方式中的细节级别是由所选择的正是分析技术的目的所确定的。有些方法被精心选定以发现那些可能逃避试验的设计错误，但是另一些方法是为了保证消除某些类型的设计错误。

1.) 错误检测。错误检测的最普通正式技术被称为模式检测。这里的要求是以可决定的瞬时逻辑作为公式表达的。该元器件的模式是一个抽象的状态机器，如此设计是为了保存待测试的特性。该检测过程是自动的。一个失败的检测表示在确定了模式的元器件中有一个设计错误。失效检测的结果是一系列输入激励，该激励展示了元器件是如何不满足规定的要求的。  
(2.) 错误消除。预防错误的正式方法通常是根据一个规范表述语言、用支持检测理论为

基础的。由于有了不断增强表达方法，再复杂的要求也可以陈述，再详细的元器件模式也能组建。不过，检测过程只能部分自动化。元器件模式的正确详细级别可能是一个合成HDL描述。在某些情况下，相同的模式可以用作激励和正是分析。一个完成了的检测是元器件逻辑正确、规定的要求可用于已经分析的输入空间的证明。

# 3.3.3.2 正式方法分析

演绎正式分析有三种可能的结果：

1.) 如果检测成功，则该鉴定活动是完整的。设计安全的级别取决于所使用模式的正确性。例如，如果硬件项目的模式对应于一个详细设计，则该检测提供了相当于排气试验的功能正确性保证。  
2.) 在某些情况中，一个失败的检测会产生一个明显的反例，也就是说，它会标识一个试验情况来特别说明该设计是如何不满足规定的要求。这可以表明一个设计缺陷或要求缺陷。这样的缺陷可以通过改正设计、修改要求来解决。这表明它不是一个物理上可以实现的条件，否则用其它的方法。所有的反例都应改标识以逐个解决。对设计或要求的变化需要考虑以前的过程。

a. 当一个设计或要求已经被改变、以处理失败检测所标识的缺陷时，应再次检测以肯定所做的修改已经成功解决了所标识的问题。重复该循环直到获得成功的检测。  
b. 万一反例被认为解决而没有改变要求和设计、但改工具只标识了一个反例，即被解决了的反例，则该过程应该被标识以使它能够标识所有另外的反例。  
3.) 要解决的最困难的事情是不能做检测也不能标识一个反例。为了简化该鉴定过程，一个可能的选择是修改设计。另外，该鉴定活动可以用检验所处理的情况与其他方法所处理的情况之间的清楚的草图来分解。对该设计和所获得的要求的改变应当回头考虑FFPA。

# 3.3.3.3 正式方法数据

在正式方法应用期间所产生的数据包括：

1.) 所使用的特殊正式方法和元器件或正式方法所应用的 FFP 的描述。  
2. 要求的正式陈述。  
3）元器件的正式模式。  
4.) 检验或产生检验的详细资料与元器件和要求的正式陈述有关，同时包括溯源数据的相互关系。  
5.) 所使用的工具的标识和工具检查结果。  
6.) 作为分析结果所增加和修改的鉴定试验情况和要求的标识。  
7.) 通过分析而获得的用于 FFP 的完整鉴定级别的说明。包括分析不一致性的清单。为了接受不一致性, 对鉴定试验或有关要求及其基本原理的改变并未解决分析部一致性问题。

# 附录C 术语汇总

下面的术语在本资料中使用。没有在本附录中定义的术语可能已在有关的文本中定义。验收：认证机构承认，所提供的数据、论点或相当的观点，满足应用要求。

适航性：某产品的条件，该产品可以是飞机、飞机系统或元器件。在该条件下产品可以安全地操作，从而完成预期的功能。

分析：一个数学或其它逻辑推理的过程。它可以在一定的前提下得出有关设备或硬件特殊能力的结论，或某一特殊应用的充分性。

异常行为：与特殊要求不一致的行为。

申请者：请求认证当局批准的个人或组织。

专用集成电路（ASIC）：所研制的用来完成某一功能的集成电路。它包括(但不限于)：门阵列、标准电池以及包含线性、数字式和混合方式技术的所有定制元器件。

批准：表达同意某一观点或给予正式批准的证书。

装配：许多元器件或其任何组合，安装在一起用来完成某一特殊功能。

评估：根据工程判断给出的估价。

假设：所提供的没有证据的观点和原理。

安全保证：必要的有计划的结果和系统的操作。它提供充分的信心和证据，以表示某产品或过程满足给定的要求。

可利用性：某产品或功能处于可操作状态的概率。

基线：某个已标识和批准的配置，它可以作为进一步设计的基础，并且只能通过改变控制过程来改变。

认证：由认证当局给出的法律承认。即某个产品、服务、组织或个人符合一定的要求。这样的认证要求技术上检查该产品、服务、组织或个人，并且通过发布证书、批准或其它文件以正式承认符合应用要求，这是国家法律和法规所要求的。尤其，产品的认证涉及：

a. 评估产品的设计以保证该产品符合一定的标准。该标准适用于该类型的产品，结果表明符合可接受的安全级别。  
b. 对单个产品进行评估以保证它符合所认证类型的设计。  
c. 根据上述两点，国家法律要求证书发放要与有关标准一致。认证当局：根据一定的要求，国内负责发放证书的组织或个人。

注：有关飞机、发动机或螺旋桨类认证，或有关设备批准之类的事情，应该由认证当局来处理。与连续适航性有关的事情应该由适航性管理当局处理。

认证基础：由认证当局规定，同时与申请者协商，作为特殊认证要求及补充已颁布条例的特殊条件。它们是飞机、发动机及螺旋桨认证基础。

认证信誉：由认证当局检验某工艺、产品和演示满足认证要求。

变化控制：（1）记录、评估、批准或不批准以及配置正式确立后对配置项目所作的协调变化的过程，或者基线确立后对基线所做的协调变化的过程。（2）系统的评估、协调、批准或不批准以及配置实体确立后对一个配置项的配置所实施的变化的批准，或配置实体确立后对基线所做的改变。

商用现成（COTS）元器件：由供应商研制的用于多用户的元器件、集成电路或子系统。其设计和配置由受供应商的规范或工业规范提供控制。

注：COTS 元器件，可以包括电阻器、电容器、微处理器、未设置的现场可编程门阵列和可擦除可编程逻辑设备、其它集成电路类型及其可实现模式、印刷电路和可从供应商处获得的作为目录项的完整的外场可更换件。

普通故障模式：可引起2个或更多项目、子项目或功能异常的事件。

复杂硬件项目：不简单的所有项目都被认为是“复杂”。见“简单硬件项目”的定义。

一致性：强制机制的优良性能，期待或规定结果与实际结果之间的一致性。

元器件：一个自包含部件、组合部件、分组件子装配或完成一个系统不同功能的单元。

元器件的降格使用：这是一种通过强迫改变元器件使用极限来增加元器件操作裕度的设计方法。该元器件使用极限比通常或制造商规定的元器件操作额定值更严格。

并行工程：为了保证每个原理的特殊要求都被考虑到，多个原理都参与硬件设计过程。

配置：完整规定了实现某个功能的配置项目列表。

配置标识：规定或指定一个配置项的过程。

配置一致性：对一个配置项目或一个配置给出的唯一名字作为配置标识的结果。

配置项目：一个或多个元器件、工具或数据项作为配置管理的单元。

配置管理：（1）配置标识的过程、问题的控制和配置一致性的改变。（2）使用技术和管理方法以及监控手段来标识和记录一个 配置项的功能及物理特性，以及对这些特性的变化控制，然后记录并报告变化控制过程和实现状态。

一致性：根据某标准、规范或图示所建立的正确一致性。

一致： 硬件项目根据规定的文本与物理实现的一致性。

有效范围分析：确定提出的硬件鉴定过程满足其目标的过程。

缺陷：某特性与规定要求的不一致性。

获得的要求：由硬件设计过程获得的附加要求，但该要求不能直接溯源到更高级的要求。

设计安全保证：在适当的可信度，所有计划的系统的操作都用来实现标识并改正设计错误，使得硬件满足应用鉴定基础要求。

设计裕度分析：各种硬件元器件设计裕度的总效应使得某产品满足或超过其性能要求以及生产性和维护性要求。确定这些要求的过程被称为设计裕度分析。

设计过程：用下列过程，根据一系列要求建立硬件项目的过程：要求抓住概念设计、细节设计、实现及生产过渡。

设计工具：工具的输出是硬件设计的一部分，由此产生了误差。例如，ASIC路由器或根据草图、其它要求设计电路板或芯片的工具。

漏译：错误：——在要求、设计或工具中的出错。

相当类别：某功能输入空间的分割，使得某类代表值的测试相当于其它类值的测试。

暴露时间：最后得知硬件项目正常操作和得知再次正常操作之间的时间周期。

失效：在规定极限内一个系统或系统元器件不能完成要求功能。当遇到故障时，即产生失效。

失效条件：由一个或多个失效所引起的对飞机或其设备的直接或间接影响。这些影响不利于操作并与环境条件有关。

失效影响：（1）作为失效的结果，对一个项目操作的描述。

（2）失效方式对操作、功能、某系统或某项目的状态的影响。

失效方式：某项目发生故障的方式。

失效率：在某一项目范围内所发生的故障总数，除以规定条件下该项目工作的总时间。

故障：（1）由于误差或随机事件，硬件表现出的错误。如果发生故障，则可产生失效。（2）一个项目产生不需要的异常。

试制品：提供的用于检查、鉴定生产图纸、工具和过程的单元。

试制品的检验：鉴定某硬件符合制造过程规范。操作生产硬件项目（代表未准备好配置）作为生产批准的先决条件。

功能缺陷：可引起硬件功能异常，即使没有发生硬件物理失效。硬件产生的操作异常反过来引起有关的软件功能异常。

功能失效路径：特殊的相互依靠的电路会引起实现某功能硬件或与某功能有关的硬件中异常现象。

功能路径：特殊的实现某功能的相互依赖电路。

小故障：一个输入转变或电压尖峰发生的时间小于受影响逻辑延迟的时间。该逻辑会产生输出。

指导：为了符合鉴定要求，咨询有关专家。

硬件设计寿命周期过程：由某组织决定的一套设计或支持过程。它足以设计出一个硬件项目。

硬件描述语言：HDL在本资料中用来代表所有的硬件描述语言。它包括“Verilog HDL”、高速集成电路硬件描述语言和模拟硬件描述语言。

硬件项目：物理存在的项目。它通常指LRU、电路板组件、电源和元器件。

硬件分割：通过硬件的物理分割和隔离来加强可靠性和安全性的方法。硬件用来实现该功能（包括冗余度），从而避免由于普通故障产生失效影响。

硬件/软件结合：硬件和软件的结合来实现某一应用和功能。

独立性：保证目标评估的责任分工。参考智能独立，如单个独立，而非部门或公司的独立。

1.) 为了鉴定，通过对技术数据正确性的评估来获得独立性。数据的获得方法是通过技术人员或某设备（而不是那些用来产生数据的设备）。  
2.) 为了过程安全，通过对过程一致性的评估来获得独立性。一致性的评估由人或某设备进行（而不是用完成该过程的设备

来进行）。

实现：由某规范产生物理现实的操作。

检验：对所提供的设备和服务的检测，包括何时提供原材料、元器件、中间装配和服务以确定它们是否满足特定的要求。

集成电路：由分别关联的、在原来位置上或单个层面内形成的元件所组成的电路（它用来完成某一个电子功能）。

完整性：某项目用来完成特定功能的属性。

项目：一般意义上的项目指的是硬件元器件、系统或软件。

寿命周期：开始硬件设计（或改变）与设计（或改变）完成（即完成到生产的转换）之间的时间。

注：在本资料中，除非另有规定，否则这里的寿命周期指的是“硬件设计寿命周期”。

可维护性：当维护是根据规定的过程和方法完成时，设计和安装的特性可以表达为在给定的时间内某项目保持或恢复到规定的条件的概率。

功能失效：在某条件下，某项目在规定极限之外操作。

制造性：通过优化材料及制造工具或通过使用设计技术（极大地减小元器件变化对功能性的影响），产品特性有利于批量生产。

一致性方法：用来满足鉴定规范中规定的飞机或发动机要求的方法。实例包括陈述、图分析、计算、试验、模拟、检查和环境合格。如果合适，可使用鉴定当局所发布的咨询材料。

监控：（1）安全。在某系统内用来检测该系统异常行为的功能。

(2) 过程安全。目睹或检查所选择设备的测试或其它活动的记录以保证该活动在控制下。报告的结果代表了预期的结果。监控通常与延时情况下所做的活动有关, 那里  $100\%$  目睹实不可行或不必要的。监控允许检验所要求的活动按计划进行。

过应力缺陷：起元器件超过额定设计极限或在硬件设计寿命周期内遇到过大的应力。

零件号：用来标识某个配置项目（即配置实体）的一组数字、字母或其它字符。

计划过程：规定和协调硬件设计活动的过程和支持过程。

系统安全基本评估：根据功能的不利性评估和故障条件分类，对系统结构及其实现所进行的系统性的评估，从而确定系统结构中所有项目的安全要求。

注：一个系统的基本安全评估适用于开发中的系统。它可以用来直接进行进一步的安全分析以完成系统安全评估。

过程：一组相关联的活动，用于产生规定的输出或产品。

过程安全：过程安全的目的是保证计划的实施，同时满足硬件设计寿命周期目标并完成这些活动。

产品：硬件、软件、项目或系统对规定要求的响应。

产品服务经验：在某周期内硬件在已知的环境内操作，且在该周期内故障已被记录下来。

生产：通过规定的和控制的过程所进行的产品制造。

可编程逻辑设备（PLD）：作为电子元器件购买并改变以完成特殊的应用功能的元器件。PLD包括（但不限于）：可编程阵列逻辑元器件、可编程逻辑阵列元器件、通用阵列逻辑元器件、现场可编程门阵列元器件和可擦除可编程逻辑设备。

样机：生产前的硬件项目。它使用检验过的元器件，完全代表了最后的产品。它完全适合形式、设计和性能的评估。

投入使用：正式地将硬件项目的数据置与配置控制下。

可靠性：在规定条件下，某硬件项目以一定的时间间隔完成预定功能的可能性。

可靠性缺陷：当处于不超过额定极限的应力条件时，由于过高的速率引起硬件故障的缺陷。应力过大缺陷和可靠性缺陷都可能表现为过高的随机故障率、过高的早期故障率和过高的磨损率。

要求：可鉴定规范的可标识元件。

可逆工程：在特殊环境内通过研究硬件结构、功能和性能而再次实现硬件项目。

评估：进行定量分析以评估该计划、要求、设计数据、设计概念或设计实现以展示高的可信度：这些要求已经满足或将要满足。

危险：频率的组合以及指定的有害状态的后果。

鲁棒性缺陷：当处于应力条件并且使用寿命不超过设计极限时，所引起的失效或操作不正常。这些缺陷的结果可能包括对环境应力的敏感性和使用寿命的不稳定性。

安全性：在此状态下的危险低于临界危险。临界危险是可接受危险的上限。对技术过程或状态尤其如此。该危险由发生的概率或几率以及可能的损害或伤害来定义。

相似性：适用于可比较特性的系统和已由申请者鉴定的在飞机上使用过的系统。可以进一步肯定，主系统中没有哪个部分由于环境和安装更危险，并且操作应力决不比类似的系统更严厉

仿真器：是一种在硬件鉴定期间所使用的设备、计算机程序或系统。它接受相同的输入并且在给定的系统中产生相同的输出。

软件：计算机程序、可能有关的资料以及与某个计算机系统操作有关的数据。

规范：规定了某硬件项目的功能和属性，一起构成某标准的要求的集合。

标准：对给定活动或规定数据项的内容提供指导和评估的比较基础和规则。

结构：形成整体的各部分的特殊安排和相互关系。

支持过程：用来支持设计过程的过程。它由下列过程之一组成：有效性、鉴定、配置管理、过程保证和鉴定关系。

系统结构：用来选择实现系统要求的硬件和软件的结构。

系统：组织起来完成某一特殊功能或某一组功能的硬件和软件的总和。

系统安全评估（SSA）：对指定的系统进行系统的、综合的评估以表示满足相关的安全要求。

试验：用规定的客观标准、用“通过”或“失效”结果对该过程进行定量的分析以证明其性能。

- 硬件项目。在控制条件下工作时，决定其性能特性。  
- 电子数字计算。肯定某元件、元器件、程序等的状态或条件。  
$\bullet$  有时用作一般术语，包括检查和诊断过程。  
$\bullet$  大概与检查相同。  
- 是检查的元件并且通常表示由提供的元件特性的技术方法确定，建议包括功能操作。它涉及到使用建立的科学原理和过程。

可测试性：（1）充分地测试硬件项目以保证硬件项目的所有状态都能完成其功能的能力。（2）硬件项目能容易地被测试，以提供满足要求的证据。

试验： 验证某硬件项目性能的过程。

试验过程：用于给定试验控制条件的详细指令。

工具评估：一组活动，用来评估硬件项目的设计和鉴定中所使用的工具可以提供证据，即该工具能够正确完成其功能，并且该功能与硬件项目完成的该功能的设计安全级别一致。

工具合格鉴定：在一个特殊机载系统内，某工具所获的鉴定认证的必要过程。

溯源性：在硬件项目或过程之间可标识的联系。例如，在一个要求和要求源之间或在一个鉴定方法及其基本要求之间。

不稳定性：由外部事件，如闪电或其它环境事件所引起的干扰。

有效性： 确定那些要求是正确的要求、那些要求是不完整的过程。

鉴定： 对要求的实现所出评估以确定满足这些要求。

鉴定工具：用来保证性能满足预先确定的标准和要求的工具。这些工具不会产生误差，但是会检测不出误差。例如，能测试出实际电路性能的模拟电路和数字电路仿真器或自动化测试。
